Chapter 01

The Storage Protocol — Connective Tissue

The three structures in this post — LRU Cache, TTL Key-Value Store, and generator pipeline — look independent. They share a deeper connection: one of them can back another, and the mechanism that enables this is a structural protocol.

Python's Protocol is structural typing. Unlike an abstract base class, a Protocol doesn't require explicit inheritance — any class that implements the right methods satisfies it automatically. This is duck typing with a static type-checker opt-in.

from typing import Protocol, runtime_checkable, Iterable

@runtime_checkable
class Storage[KT, VT](Protocol):
    def __setitem__(self, key: KT, value: VT) -> None: ...
    def __getitem__(self, key: KT) -> VT: ...          # raises KeyError on miss
    def __delitem__(self, key: KT) -> None: ...
    def __contains__(self, key: object) -> bool: ...
    def __len__(self) -> int: ...
    def keys(self) -> Iterable[KT]: ...

This is exactly the interface a Python dict satisfies. And @runtime_checkable makes that verifiable at runtime — not just at type-check time:

print(isinstance({}, Storage))  # → True

The payoff comes later. The TTL Store's constructor takes a Storage[KT, CacheEntry[VT]]. That means it can accept a plain dict or an LRUCache — as long as the cache also satisfies the protocol. Building the LRU Cache to satisfy Storage is Chapter 02's goal. Using that to compose the two into one system is Chapter 05's.

An abstract base class (ABC) requires explicit class MyStore(ABC) and @abstractmethod. A Protocol requires nothing — any class with the right methods qualifies. The tradeoff: ABCs catch missing methods at class definition time; Protocols catch them at type-check time (mypy) or not at all without @runtime_checkable. Protocols are the right choice when you want to retrofit an interface onto types you don't control — like dict.

Chapter 02

LRU Cache — O(1) and Protocol-Compliant

An LRU Cache keeps the N most recently accessed entries. On capacity overflow, the least recently used entry is evicted. The constraint is that both get() and put() must be O(1) — which rules out any approach that involves scanning.

The O(1) Invariant

The solution is two structures working in concert: a dict for O(1) key lookup, and a doubly-linked list for O(1) recency ordering. The dict maps each key to its node. The list maintains a strict invariant: head.next is the LRU entry, tail.prev is the MRU entry. Every access moves its node to the MRU position; every eviction removes from the LRU position.

head ←→ [A] ←→ [C] ←→ [B] ←→ tail
         LRU              MRU

dict = { "A": <node_A>, "C": <node_C>, "B": <node_B> }

get("C"):  unlink C, relink before tail  →  head ←→ [A] ←→ [B] ←→ [C] ←→ tail
put("D"):  add D before tail, evict head.next (A)  →  [B] ←→ [C] ←→ [D]

Node and Deque — Keeping Pointers Symmetric

The assign_next method on Node is the structural linchpin: it sets both the forward pointer on the caller and the backward pointer on the target in a single call. Every pointer operation goes through it, so next and prev are always symmetric. A broken invariant cannot be silently introduced.

class Node[KV, VT]:
    def __init__(self, key: KV, value: VT):
        self.key, self.value = key, value
        self.next: Node[KV, VT] | None = None
        self.prev: Node[KV, VT] | None = None

    def assign_next(self, node: Node[KV, VT]) -> None:  # O(1)
        self.next = node
        node.prev = self


class Deque[KV, VT]:
    def __init__(self):
        self._head = Node[KV, VT](None, None)  # LRU sentinel
        self._tail = Node[KV, VT](None, None)  # MRU sentinel
        self._head.assign_next(self._tail)

    def remove(self, node: Node[KV, VT]) -> None:       # O(1)
        node.prev.assign_next(node.next)
        node.prev = node.next = None

    def add_to_end(self, node: Node[KV, VT]) -> None:   # O(1)
        self._tail.prev.assign_next(node)
        node.assign_next(self._tail)

    def move_to_end(self, node: Node[KV, VT]) -> None:  # O(1)
        self.remove(node)
        self.add_to_end(node)

    def pop_front(self) -> Node[KV, VT]:                 # O(1) — evicts LRU
        node = self._head.next
        self.remove(node)
        return node

Implementing the Storage Protocol

For LRUCache to be usable as a backing store for KVStore, it must satisfy Storage[KV, VT]. That means implementing the dict-protocol dunder methods alongside the cache-friendly get(key, default) API.

The two APIs serve different callers. Cache clients want get(key, default=None) — a miss returns a default, not a crash. The Storage protocol wants __getitem__ — a miss raises KeyError, consistent with how dict behaves. Both coexist; neither replaces the other.

import threading

class LRUCache[KV, VT]:
    def __init__(self, capacity: int):
        if capacity <= 0:
            raise ValueError(f"capacity must be > 0, got {capacity}")
        self._capacity = capacity
        self._cache: dict[KV, Node[KV, VT]] = {}
        self._deque: Deque[KV, VT] = Deque()
        self._lock = threading.Lock()

    def _promote(self, key: KV, value: VT) -> None:
        # Internal: upsert + move to MRU. Must be called under self._lock.
        if key in self._cache:
            self._deque.remove(self._cache[key])
        node = Node(key, value)
        self._deque.add_to_end(node)
        self._cache[key] = node
        if len(self._cache) > self._capacity:
            evicted = self._deque.pop_front()
            del self._cache[evicted.key]

    # ── Cache API ──────────────────────────────────────────

    def put(self, key: KV, value: VT) -> None:                        # O(1)
        with self._lock:
            self._promote(key, value)

    def get(self, key: KV, default: VT | None = None) -> VT | None:  # O(1)
        with self._lock:
            if key not in self._cache:
                return default
            self._deque.move_to_end(self._cache[key])
            return self._cache[key].value

    # ── Storage Protocol ───────────────────────────────────

    def __setitem__(self, key: KV, value: VT) -> None:              # O(1)
        with self._lock:
            self._promote(key, value)

    def __getitem__(self, key: KV) -> VT:                             # O(1)
        with self._lock:
            if key not in self._cache:
                raise KeyError(key)
            self._deque.move_to_end(self._cache[key])
            return self._cache[key].value

    def __delitem__(self, key: KV) -> None:                          # O(1)
        with self._lock:
            if key not in self._cache:
                raise KeyError(key)
            self._deque.remove(self._cache[key])
            del self._cache[key]

    def __contains__(self, key: object) -> bool:                      # O(1)
        with self._lock:
            return key in self._cache

    def __len__(self) -> int:                                          # O(1)
        with self._lock:
            return len(self._cache)

    def keys(self) -> list[KV]:                                        # O(N) snapshot
        with self._lock:
            return list(self._cache.keys())

# Structural conformance — no inheritance needed:
print(isinstance(LRUCache(10), Storage))  # → True

Both put() and __setitem__ do exactly the same thing. Rather than duplicating the logic or calling one from the other (which would acquire the lock twice), the shared logic lives in _promote() which is called under the lock by both. This is the correct way to factor out lock-protected shared logic in Python — never call a public method from another public method if both acquire the same non-reentrant lock.

Thread Safety — Why a Single Lock Is Enough

The dict and Deque must always agree. A get() that moves a node in the deque after finding it in the dict, or a put() that inserts into the deque but fails before updating the dict — either leaves the cache in a broken state. A single threading.Lock() wrapping each public operation makes both structures update atomically.

For read-heavy workloads, a read-write lock (threading.RLock is not the same thing — Python has no stdlib RWLock) could improve throughput by allowing concurrent reads. But it adds complexity: get() also writes (it moves the node to MRU), so it cannot hold only a read lock. In practice, a plain Lock and appropriately sized capacity is the right starting point.

The Test Invariant That Bites

The one invariant that consistently trips people up: updating an existing key must not trigger eviction of a different key. Consider capacity=2, keys A and B both present. Calling put("B", new_value) 200 times must not evict A — it's a re-promotion to MRU, not a new entry. The test that catches this:

cache = LRUCache[str, str](2)
cache.put("A", "a")
cache.put("B", "b")
for i in range(200):
    cache.put("B", str(i))
assert cache.get("A") == "a"   # must still be present
assert cache.get("B") == "199"  # latest value
Chapter 03

TTL Key-Value Store — Heap-Based Lazy Deletion

LRU evicts by access pattern — unused entries get pushed out. TTL evicts by time — entries expire after a fixed duration regardless of how often they're accessed. A configuration value read on every request should still expire after 5 minutes even though it's always the MRU entry. The two policies are not interchangeable and should not be conflated.

The Data Model — Frozen Dataclasses

Cache entries are frozen dataclasses. frozen=True makes the instance immutable after creation: Python generates a __hash__ and overrides __setattr__ to raise FrozenInstanceError on mutation. An entry's TTL and value cannot be accidentally modified through an alias.

from dataclasses import dataclass, field
from datetime import datetime, timedelta

@dataclass(frozen=True)
class MetaData:
    expires_at: datetime

@dataclass(frozen=True)
class CacheEntry[VT]:
    value: VT
    metadata: MetaData

# Usage: wrap any value with a TTL
entry = CacheEntry(
    value="api-result",
    metadata=MetaData(expires_at=datetime.now() + timedelta(seconds=30))
)

The Problem with a Sorted Deque

A deque sorted by expires_at is the intuitive approach. The front holds the soonest-to-expire entry; sweeping expired entries is just popping from the front. But put() with an existing key is a problem. You must find and remove the old entry from the middle of the deque — O(N) — then find the correct sorted position for the new entry — also O(N). Updates are expensive.

Worse, the linear scan across all positions uses a counter-loop because deque doesn't support binary search. The actual implementation ends up with two O(N) passes on every update — exactly the wrong tradeoff for a structure that's supposed to be fast at writes.

The Min-Heap with Version-Based Lazy Deletion

A min-heap gives O(log N) push. The problem it introduces is removal: Python's heapq has no O(log N) deletion by value. When a key is re-inserted with a new TTL, the old heap entry becomes stale but cannot be efficiently removed.

The solution is lazy deletion: don't remove stale entries immediately. Leave them in the heap. When we sweep on get(), check whether each popped entry is still the current version for its key. If not, discard it. This requires a monotonic version counter and a version registry.

@dataclass(frozen=True, order=True)
class _HeapEntry[KT]:
    expires_at: datetime               # primary sort key
    version: int                       # secondary sort key; also the stale-check discriminator
    key: KT = field(compare=False)   # excluded from heap ordering

dataclass(order=True) generates __lt__ and friends using fields in declaration order, skipping those marked compare=False. This is exactly what heapq needs. The version field as the secondary key breaks ties when two entries share an identical expires_at — without it, Python would attempt to compare KT values, which may not support ordering.

import heapq

class KVStore[KT, VT]:
    def __init__(self, store: Storage[KT, CacheEntry[VT]]):
        self._store = store
        self._heap: list[_HeapEntry[KT]] = []
        self._version: dict[KT, int] = {}  # key → active version
        self._clock = 0                    # monotonic counter

    def put(self, key: KT, entry: CacheEntry[VT]) -> None:   # O(log N)
        self._clock += 1
        self._version[key] = self._clock
        self._store[key] = entry
        heapq.heappush(
            self._heap,
            _HeapEntry(entry.metadata.expires_at, self._clock, key)
        )
        # The old heap entry for this key (if any) is now stale.
        # It will be identified and discarded by _sweep().

    def get(self, key: KT) -> CacheEntry[VT] | None:          # O(k log N) amortized
        self._sweep()
        return self._store[key] if key in self._store else None

    def _sweep(self) -> None:
        now = datetime.now()
        while self._heap:
            top = self._heap[0]
            if top.expires_at > now:
                break                          # heap min is not expired — stop
            heapq.heappop(self._heap)

            if self._version.get(top.key) != top.version:
                continue                       # stale entry from an overwrite — skip

            if top.key in self._store:
                del self._store[top.key]
            del self._version[top.key]

Why the Version Discriminator Is Necessary

Without the version check, consider this sequence: put("k", ttl=5s) at t=0, then put("k", ttl=60s) at t=1. The heap now contains two entries for "k": one expiring at t=5, one at t=61. At t=6, _sweep() pops the first entry. Should it delete "k" from the store?

Without a version check, the answer is yes — and that's wrong. The entry at t=61 is still valid. The version check resolves this cleanly: the stored version is 2 (set by the second put), the popped heap entry carries version 1. They don't match — this is a stale entry, skip it.

Using expires_at as the discriminator instead of version would work in most cases, but breaks when two puts happen close enough together that floating-point datetime precision produces identical timestamps. The version counter is a strictly monotonic integer — it cannot collide.

OperationSorted DequeMin-Heap + Lazy Delete
put() — new keyO(N) sorted insertO(log N) heappush
put() — existing keyO(N) find + O(N) re-insertO(log N) heappush; old entry left stale
get() — sweep costO(k) — k expired entries at frontO(k log N) — k expired + stale pops
SpaceO(N)O(N + S) — S = stale entries pending sweep

The heap approach trades a slightly higher sweep cost (log N factor per expired entry) for dramatically better put() performance. For systems where writes are frequent and keys are frequently re-set with new TTLs, this is the right trade.

Stale entries accumulate in the heap between sweeps. If put() is called far more often than get(), the heap can grow to O(total puts) rather than O(live keys). A simple heuristic: if len(heap) > 2 × len(store), run a full compaction pass — rebuild the heap from only the active versions. This is O(N log N) but amortized infrequently.

Chapter 04

Typed Generator Pipeline — Generator[U, bool, None]

A pipeline without flow control is a time bomb. A fast producer feeding a slow consumer accumulates items in memory silently — no error, no warning — until the process crashes or the OS starts swapping. For synchronous pipelines, Python's generator send() protocol provides a native backpressure mechanism.

Generator[YieldType, SendType, ReturnType]

Most generators are typed as Iterator[T], which only describes what they yield. The full Generator[Y, S, R] type encodes the complete contract: what the generator yields (Y), what it receives via send() (S), and what it returns when exhausted (R). When S is something other than None, the generator is a two-way channel.

from typing import Generator

# Generator[YieldType, SendType, ReturnType]
def echo() -> Generator[str, str, None]:
    received = yield "ready"   # yields str, receives str via send()
    yield f"you sent: {received}"

gen = echo()
next(gen)              # → "ready"  (primes to the first yield)
gen.send("hello")    # → "you sent: hello"

The expression received = yield value does two things: it sends value out, and when resumed via send(x), received is assigned x. You must call next(gen) before the first send(x) — the generator hasn't reached the yield yet, so there's nowhere to deposit the sent value.

Generic Producer, Transformer, and Consumer

With Python 3.12's generic function syntax, the full pipeline is typed end-to-end. The type parameters make the data flow explicit: a producer[T] feeds a transformer[T, U] which feeds a consumer[U].

from typing import Callable, Generator, Iterable, Iterator

def producer[T](items: Iterable[T], stats: dict) -> Iterator[T]:
    for item in items:
        stats["produced"] += 1
        yield item


def transformer[T, U](
    source: Iterator[T],
    predicate: Callable[[T], bool],
    transform: Callable[[T], U],
    stats: dict,
) -> Generator[U | None, bool, None]:
    for item in source:
        if not predicate(item):
            stats["dropped"] += 1
            continue
        should_drop: bool = yield transform(item)  # ← two-way channel
        if should_drop:
            stats["dropped"] += 1
    yield None  # sentinel: source exhausted


def consumer[U](
    pipeline: Generator[U | None, bool, None],
    stats: dict,
    buffer_size: int = 5,
) -> list[U]:
    results: list[U] = []
    value = next(pipeline, None)          # prime the generator
    while value is not None:
        results.append(value)
        stats["consumed"] += 1
        buffer_full = len(results) % buffer_size == 0
        value = pipeline.send(buffer_full)   # True → transformer drops next item
    return results

The Generator[U | None, bool, None] return type on transformer is precise: it yields either a transformed value or a None sentinel, receives a bool backpressure signal, and returns None on exhaustion. The | None in the yield type is the cost of using None as a sentinel — a production codebase would use a typed sentinel class instead to avoid ambiguity when U is itself nullable.

Context Manager for Lifecycle

GeneratorExit must be handled separately in the context manager. It is raised inside the generator when gen.close() is called — not an error, just early termination. Conflating it with Exception would suppress legitimate cleanup.

from contextlib import contextmanager

@contextmanager
def pipeline_context(name: str):
    stats = {"produced": 0, "consumed": 0, "dropped": 0}
    try:
        yield stats
    except GeneratorExit:
        pass                     # gen.close() called — not an error
    except Exception as e:
        print(f"[{name}] error: {e}")
        raise
    finally:
        print(f"[{name}] done — {stats}")


def run_pipeline[T, U](
    items: Iterable[T],
    predicate: Callable[[T], bool],
    transform: Callable[[T], U],
    buffer_size: int = 5,
) -> list[U]:
    with pipeline_context("main") as stats:
        source = producer(items, stats)
        pipe   = transformer(source, predicate, transform, stats)
        return consumer(pipe, stats, buffer_size)


# Example: evens squared, buffer_size=5
results = run_pipeline(
    items=range(20),
    predicate=lambda x: x % 2 == 0,
    transform=lambda x: x ** 2,
)

Using None as an end-of-stream sentinel means the pipeline cannot process items where transform(item) legitimately returns None. A typed sentinel — class _Exhausted: ...; _EXHAUSTED = _Exhausted() — eliminates the ambiguity. The consumer checks isinstance(value, _Exhausted) instead of value is None. The downside is verbosity; the upside is a pipeline that can carry any type through without special-casing.

Chapter 05

Composition — TTL Backed by LRU

The Storage Protocol was defined in Chapter 01 precisely for this moment. KVStore takes any Storage[KT, CacheEntry[VT]]. LRUCache now satisfies Storage. The composition is two lines:

from datetime import timedelta

# A TTL store whose backing memory is bounded by an LRU Cache
lru: LRUCache[str, CacheEntry[str]] = LRUCache(capacity=500)
store: KVStore[str, str] = KVStore(store=lru)

assert isinstance(lru, Storage)   # True — structural typing at work

# Two eviction policies, one system:
# LRU evicts entries beyond capacity=500 (access-order policy)
# TTL evicts entries after their expiry (time policy)
store.put("config:rate_limit", CacheEntry(
    value="1000/min",
    metadata=MetaData(expires_at=datetime.now() + timedelta(minutes=5))
))

This works without any adapter code because LRUCache.__setitem__, __getitem__, __delitem__, __contains__, __len__, and keys() are all present and carry the right signatures. The Storage Protocol makes this verifiable statically and at runtime.

Plugging the Pipeline In

The store and the pipeline connect naturally. A typed producer pulls from the KVStore, calling an upstream fetch on cache miss and caching the result. The consumer drives it through send()-gated backpressure. The entire system is typed end-to-end.

def cached_producer[V](
    keys: Iterable[str],
    store: KVStore[str, V],
    ttl: timedelta,
    fetch: Callable[[str], V],
    stats: dict,
) -> Iterator[V]:
    for key in keys:
        cached = store.get(key)
        if cached is None:
            value = fetch(key)
            store.put(key, CacheEntry(value, MetaData(datetime.now() + ttl)))
        else:
            value = cached.value
        stats["produced"] += 1
        yield value


# Wire it all together
lru   = LRUCache[str, CacheEntry[str]](capacity=500)
store = KVStore[str, str](store=lru)

with pipeline_context("fetch") as stats:
    source = cached_producer(keys, store, ttl=timedelta(seconds=30), fetch=api_call, stats=stats)
    pipe   = transformer(source, predicate=is_valid, transform=normalize, stats=stats)
    output = consumer(pipe, stats, buffer_size=10)

Each layer owns exactly one axis of control. LRU owns capacity. TTL owns freshness. The pipeline owns throughput. None of them knows about the others. The Storage Protocol is the only shared contract — and it's structural, so adding a fourth layer (say, a write-through cache) requires no changes to any existing type.

The value of a protocol is not in what it forces you to implement — it's in what it lets you swap out later without rewriting anything upstream.
