The Baseline · Relational

PostgreSQL Internals

Every NoSQL decision is a deliberate trade-off against what Postgres gives you. You must understand Postgres's architecture first — or you cannot reason about why the trade-off is worth making.

01

The B-Tree: Why It Governs Everything

🌳
B-Tree Index Structure
The index type behind every PRIMARY KEY, UNIQUE, and default CREATE INDEX
Core ▼
What it is
A B-Tree is a self-balancing tree where every node holds between d and 2d keys (typically 100–400 keys per node in Postgres, sized to fit a single 8KB page). Leaf nodes hold the actual index entries: (key_value → heap_tid) where heap_tid is a physical pointer to the row's location in the heap file (page number + slot offset). Internal nodes hold separator keys + child page pointers.

Why this gives you O(log N) reads and O(log N) writes
Tree height = logd(N). With d≈300 and 100M rows: height = log₃₀₀(100M) ≈ 4 levels. Every lookup = 4 page reads (each may be a disk seek). The leaf-to-leaf linked list enables efficient range scans — after the first binary search descent, subsequent rows are sequential page reads.

The write cost: Every INSERT/UPDATE/DELETE must update the index page. If the page is full, it splits into two pages — the parent must be updated too. This cascades upward (rarely to root). All this happens synchronously before the transaction commits. This is the fundamental reason writes in Postgres are slower than in an LSM-tree engine.
🔑 The Core Constraint
B-Tree writes are in-place updates on existing pages. To maintain sort order, an insert into a full page requires splitting. This means random I/O even for sequential inserts. An LSM-tree (Cassandra, RocksDB) converts all writes into sequential appends — that's the fundamental architectural difference, not "NoSQL is faster."
📄
Heap File + MVCC
How rows are actually stored and how transactions work
Storage ▼
Heap file layout
The actual table data lives in a heap file — a flat array of 8KB pages. Each page has a header, then row slots stored from the end backward, and free space in the middle. Each row (called a tuple) has a 23-byte header containing: xmin (transaction ID that created it), xmax (transaction ID that deleted it), ctid (physical location of this tuple), null bitmap, and alignment bytes. Then the actual data.
MVCC: why UPDATEs write new rows
Postgres implements isolation via Multi-Version Concurrency Control. When you UPDATE a row, Postgres does NOT modify the existing row in-place. It writes a new version of the row (with the new values and a new xmin) and marks the old version with xmax = current_txn_id. Both versions exist on disk simultaneously.

A concurrent reader with an earlier transaction snapshot still sees the old version. When the transaction commits, new readers see the new version. VACUUM periodically removes dead tuples (rows where xmax is a committed transaction and no active snapshot can see them anymore).

The implication: Every UPDATE is actually a write of a full new row + an index entry update for every index on the table. A heavily-updated wide table with many indexes is expensive. This is called write amplification — one logical UPDATE becomes N physical writes (N = number of indexes).
⚠ Interview Insight
HOT (Heap-Only Tuple) optimization: if the updated column is not indexed and there's room on the same page, Postgres chains the new tuple to the old one without updating any index. This is why "never index columns you update frequently" is real advice — it enables HOT and dramatically reduces write amplification.
📝
WAL — Write-Ahead Log
The durability mechanism that everything else is built on top of
Durability ▼
What the WAL is
Before any heap page or index page is modified, Postgres writes a record of the change to the Write-Ahead Log — a sequential append-only file. The rule: log record must be flushed to durable storage before the data page is considered written. On commit, Postgres calls fsync() on the WAL, not on the heap/index pages. The heap pages are written lazily by the background writer.

On crash recovery: Postgres replays WAL from the last checkpoint. Any heap/index page changes that weren't flushed are reconstructed from WAL records.

Why WAL is the write bottleneck at scale
All writes from all connections share a single WAL stream. There is one WAL writer process. Every committed transaction must wait for its WAL records to be fsynced. At extreme write rates (~100K+ rows/sec from hundreds of connections), WAL becomes the serialization point. wal_writer_delay, synchronous_commit = off, and commit_delay are tuning knobs, but you cannot fundamentally parallelize the WAL — it's an intentionally sequential log. This is why Cassandra's commit log per-node + parallel coordinator model scales writes differently.
🧠
Query Planner + Statistics
The cost-based optimizer that chooses between seq scan, index scan, hash join, merge join
Optimizer ▼
How the planner decides
Postgres maintains statistics per column in pg_statistic: null fraction, distinct count, most common values + their frequencies, and a histogram of value distribution. ANALYZE populates these (default: ~300 histogram buckets). The planner uses these statistics to estimate how many rows a predicate will return — and picks the access method with the lowest estimated total cost (random I/O cost × pages + CPU cost × rows).
When it goes wrong — and why this matters for NoSQL
The planner cannot estimate cardinality for JSONB nested paths — data->'specs'->'weight' has no histogram. It assumes a default 0.5% selectivity, which is almost always wrong. Bad estimate → wrong plan → full seq scan on a 50M-row table when an index would help.

For multi-column predicates with correlated columns (e.g., city + zip code), the planner assumes independence and underestimates the correlation — leading to over-estimated row counts and wrong join strategies. This is why ad-hoc queries on MongoDB or DynamoDB are dangerous — there is no planner equivalent to warn you.
💬 Interview Signal
Candidates who say "just add an index" without understanding selectivity, the planner's statistics, or when a seq scan beats an index scan (for >~5% of table) haven't thought about the planner. Seq scan has better sequential I/O characteristics than index scan for low-selectivity queries — the planner knows this, most candidates don't.
🔌
Process-per-Connection Model
Why Postgres needs PgBouncer and why this limits raw concurrency
Concurrency ▼
The model
Each client connection forks a dedicated backend OS process (~5-10MB RSS per process for shared memory segments + private memory). At 500 connections: ~2.5GB RAM just for process overhead, before any query data. Each process has its own local memory for query buffers, sort workspace (work_mem), and a plan cache. Context switching between 500 processes under load is measurable overhead.
Why this is the concurrency ceiling
Postgres's practical connection limit is 100–300 before you need a connection pooler (PgBouncer). PgBouncer maintains a small pool of actual Postgres connections and multiplexes thousands of application connections onto them. In transaction pooling mode, the same Postgres connection handles different client transactions back-to-back — but this breaks SET variables, prepared statements, and advisory locks if not handled carefully.

Redis, by contrast, uses a single-threaded event loop with epoll/kqueue — tens of thousands of connections share one thread, and context switching is pure user-space. MongoDB and Cassandra use thread pools with async I/O — they don't fork an OS process per connection.

What good Postgres-level thinking sounds like in an interview

Explains B-tree height and why 4 page reads = a lookup on 100M rows
"Postgres doesn't scale" without specifying which axis (write throughput? connection count? index maintenance?)
Knows UPDATE = new tuple + dead tuple, and mentions VACUUM, write amplification, HOT
"Add an index" without knowing when the planner prefers seq scan
Explains WAL as the single serialization point for writes and why it limits horizontal write scale
"Use read replicas for scale" without explaining that read replicas only help reads, not write throughput
Knows PgBouncer is needed and why: process-per-connection model vs thread/event-loop models
Jumps to NoSQL without being able to articulate what specific Postgres constraint they're working around

Wide-Column · LSM-Tree · Masterless

Cassandra Internals

Cassandra makes one fundamental trade: it sacrifices the query flexibility of a general-purpose database to achieve linear write scalability and no single point of failure. Every internals decision flows from that commitment.

01

LSM-Tree: Why Writes Are Sequential

📥
Log-Structured Merge Tree (LSM-Tree)
The storage engine that makes Cassandra's write throughput possible
Core Engine ▼
The write path — step by step
1. Commit Log (sequential append): Every write is first appended to a commit log — a sequential file on disk. This is the durability guarantee. The write is acknowledged after this flush. Cost: one sequential disk write, regardless of which key is being written. No page lookup, no B-tree traversal.

2. Memtable (in-memory sorted structure): The write is also applied to an in-memory data structure called the memtable — typically a skip list or a concurrent sorted map. Reads check the memtable first (most recent writes are here). The memtable absorbs writes until it reaches a size threshold (~64–256MB).

3. SSTable flush (sequential write): When the memtable is full, it is flushed to disk as an immutable, sorted file called an SSTable (Sorted String Table). The flush is a sequential write — the entire memtable is written in sorted key order from beginning to end. No random I/O.

SSTables and Compaction
Each SSTable is immutable — once written, never modified. Over time many SSTables accumulate for the same key ranges. A background process called compaction merges SSTables: it reads multiple SSTables sorted by key, merges them (keeping only the latest version of each row), and writes a new, larger SSTable. The old ones are then deleted.

Why this matters for reads: A read must check the memtable + potentially multiple SSTables. Cassandra uses a Bloom filter per SSTable (a probabilistic set membership test) to quickly determine if a key might be in that SSTable — avoiding unnecessary disk reads. False positives are possible; false negatives are not. This is how reads stay O(1) despite multiple SSTables.

🔑 The Key Insight
In a B-tree engine, a write to key 42 must find page containing 42 and modify it — random I/O. In an LSM engine, a write to key 42 is appended to the end of the commit log and inserted into the memtable skip list — always sequential. The sort order is established lazily at flush time, not at write time. This converts random write I/O into sequential write I/O, which is 10–100× faster on spinning disk and 3–10× faster even on SSDs.
02

Masterless Ring: How Writes Scale Linearly

💍
Consistent Hashing + Masterless Ring
Why adding a node increases write throughput proportionally
Architecture ▼
The consistent hashing ring
Every Cassandra node owns a range of a hash space (0 to 2⁶⁴). A write for partition key user_123 is hashed via Murmur3 to a token (e.g., 8,342,719,234). The node responsible for that token range receives the write as the coordinator. The coordinator then writes to N replicas (the next N nodes clockwise on the ring).

No master, no leader election. Any node can be the coordinator for any write. There is no single WAL, no single write path. 10 nodes = 10 parallel write paths. Adding a 11th node = approximately 10% more write capacity, proportionally. This is what "linear scalability" actually means mechanistically.
Replication + Consistency levels
With replication factor RF=3, every row is written to 3 nodes. The coordinator sends the write to all 3 replicas in parallel. The consistency level determines how many acknowledgments the coordinator waits for before returning success:

ONE — return after 1 replica acks. Fastest, risk: you read stale data if you hit one of the 2 non-acknowledged replicas.
QUORUM — return after ⌈RF/2⌉+1 = 2 acks. Guarantees that a QUORUM read will overlap with at least 1 write replica — strong consistency.
ALL — all 3 replicas must ack. Slowest. Availability drops if any replica is down.
⚠ The Trade-off Candidates Miss
Cassandra has no transactions across partition keys. There is no way to atomically update two rows in different partitions. Within a single partition, Cassandra supports lightweight transactions (LWT) via Paxos — but LWT has ~4× latency overhead and is rarely used at scale. If you need cross-entity atomicity, Cassandra is the wrong tool regardless of write volume.
03

Data Modeling: Designing Tables for Queries

📐
Partition Key + Clustering Key
The data model you MUST define before writing a single line of code
Data Model ▼
Partition key: the routing decision
The partition key determines which node(s) hold the data. All rows with the same partition key are stored contiguously on the same SSTable segment on the same node. A read for a partition key = one node, one SSTable range scan. This is O(1) routing — no scatter-gather.

Cardinality matters: too few unique partition keys = hot partitions (one node holds all data). Too many unique keys (e.g., using a UUID as the partition key with no clustering) = no ability to do range queries.
Clustering key: the sort order within a partition
Within a partition, rows are sorted by the clustering key and stored in that order in the SSTable. A range query on the clustering key (e.g., WHERE user_id = ? AND event_time BETWEEN ? AND ?) is a sequential read of a contiguous byte range in one SSTable file on one node. This is why time-range queries on Cassandra are fast — they're sequential I/O, not random I/O.

The constraint: You can only efficiently query using the partition key (for routing) + clustering key prefix (for range). Any other filter requires ALLOW FILTERING which is a full partition scan — equivalent to a table scan in SQL terms.
-- This is NOT schema design. This is ACCESS PATTERN design. -- The query you need: "give me user X's events in last 24h, newest first" CREATE TABLE user_events ( user_id UUID, -- partition key: Murmur3(user_id) → node event_time TIMESTAMP, -- clustering key: sorted DESC within partition event_type TEXT, payload TEXT, PRIMARY KEY (user_id, event_time) ) WITH CLUSTERING ORDER BY (event_time DESC) AND default_time_to_live = 2592000; -- Efficient: single node, sequential SSTable scan SELECT * FROM user_events WHERE user_id = ? AND event_time > now() - 24h; -- DEATH: no partition key → full cluster scan → ALLOW FILTERING SELECT * FROM user_events WHERE event_type = 'click'; -- ← never do this
💬 Interview Signal
Ask "what's your partition key?" immediately when a candidate proposes Cassandra. If they say "the primary ID," follow with: "what's the max partition size? Cassandra recommends <100MB per partition — if one user generates 10M events/day, how do you bucket?" The answer is a composite partition key: (user_id, date) — partitioning by day to cap partition size. Candidates who haven't thought about partition sizing haven't designed for Cassandra at scale.

What good Cassandra thinking sounds like in an interview

Explains LSM: write → commit log (sequential) → memtable → SSTable flush, never in-place update
"Cassandra has high write throughput" without explaining why
Knows Bloom filters are per-SSTable to avoid disk reads during reads
Doesn't know what compaction does or why read latency can spike during heavy compaction
Designs partition key around query, not around entity identity

Knows QUORUM read + QUORUM write = strong consistency, and can explain the math (RF=3, QUORUM=2, overlap guaranteed)
"Cassandra is eventually consistent" without knowing you can tune it per-operation

Key-Value · In-Memory · Single-Threaded Event Loop

Redis Internals

Redis's performance isn't magic — it's the consequence of three deliberate constraints: everything in RAM, a single-threaded event loop with no lock contention, and purpose-built data structures that execute in O(1) or O(log N).

01

Single-Threaded Event Loop: Why No Locks = Speed

🔄
The Event Loop Model
epoll/kqueue + single thread = zero lock contention
Core ▼
How it works
Redis runs a single main thread executing an event loop using epoll (Linux) or kqueue (BSD/macOS). The event loop multiplexes thousands of client connections in one thread. When a client sends a command, the kernel notifies Redis via epoll that data is ready. Redis reads the command, executes it against the in-memory data structure, and writes the response — all in the same thread, atomically.

Because there is exactly one thread touching any data structure at any time, there are zero mutex locks on the data path. A GET command is literally: hash table lookup → return value. No lock acquire, no lock release. On a modern CPU, this path executes in ~100–200 nanoseconds.
Why single-threaded doesn't mean slow
The bottleneck for a Redis operation is not CPU — it's network round-trip time (~100μs on LAN). The command itself executes in nanoseconds. A single thread saturating a 10Gbps NIC can handle ~1M simple commands/sec. Multi-threading adds lock overhead and cache invalidation traffic that for in-memory operations costs more than it saves. Redis 6+ added I/O threading for network read/write to saturate multi-core systems — but the command execution itself remains single-threaded.
⚠ The Implication Candidates Miss
Because execution is single-threaded, a slow Redis command (e.g., SMEMBERS on a set of 10M elements, KEYS *, or a Lua script with a loop) blocks ALL other clients for its duration. O(N) commands on large data structures are production incidents waiting to happen. This is why Redis documentation marks the time complexity of every command.

03

Persistence: RDB vs AOF

💾
Durability Trade-offs
What Redis actually persists and what it risks losing
Durability ▼
RDB (Snapshot)
  • Fork a child process at intervals (e.g., every 5min if 10K writes)
  • Child writes a point-in-time binary snapshot to disk via copy-on-write
  • Fast restarts — load one file
  • Risk: crash between snapshots = data loss of last N minutes
  • Fork cost: for 10GB dataset, fork is fast but CoW memory pressure spikes
AOF (Append-Only File)
  • Every write command is appended to an AOF log
  • fsync policy: always (safe, slow) / everysec (safe within 1s) / never (OS decides)
  • AOF rewrite: background process compacts log by rewriting current state
  • Risk: large AOF files, slower restarts (replay all commands)
  • Gold standard: AOF with everysec + RDB for fast restarts
🔑 Key Point
Redis with appendfsync always has similar durability to Postgres with synchronous_commit = on — every write is fsynced before acknowledgment. The difference is not durability — it's that Redis can only do this for the single-node case. Redis Cluster shards data but doesn't give you cross-shard transactions.

What good Redis thinking sounds like in an interview

Knows single-threaded means O(N) commands (KEYS, SMEMBERS on huge sets) block all clients
"Redis is fast because it's in-memory" — all in-memory DBs are fast; the architectural reason is the event loop + no locks
Picks the right data structure: ZSET for leaderboard, HyperLogLog for cardinality, Stream for fan-out events
Uses only STRING for everything — misses the entire value of Redis's native structures
Explains AOF vs RDB trade-off and can say what data loss is acceptable in their use case
"Redis can lose data" without knowing the persistence modes or how to tune them

Document Store · WiredTiger · BSON

MongoDB Internals

MongoDB's core insight is that the document is the unit of atomicity, retrieval, and schema definition. The WiredTiger storage engine gives it MVCC transactions; the BSON format and dynamic schema give it flexibility. Understanding both tells you exactly when it wins and when it doesn't.

01

WiredTiger: The Storage Engine

⚙️
WiredTiger B-Tree + MVCC
MongoDB's storage engine since v3.2 — and why it changed everything
Storage ▼
WiredTiger architecture
WiredTiger is a B-tree based storage engine with its own MVCC implementation. Each collection is stored in a WiredTiger file (essentially a B-tree of documents indexed by a monotonically increasing RecordId). Documents are stored as BSON (Binary JSON) — a binary encoding that includes type information and length prefixes for O(1) field traversal.

MVCC is implemented via a transaction history store: WiredTiger maintains old versions of modified documents in a separate in-memory structure, not in the document file itself. This is cleaner than Postgres's approach of storing dead tuples inline in heap pages — WiredTiger's history store is garbage collected separately, so there's no equivalent of Postgres's VACUUM bloat problem.
Document-level locking (vs Postgres's row-level locking)
WiredTiger uses document-level optimistic locking with MVCC. Concurrent writes to different documents in the same collection don't block each other. Two concurrent writes to the same document: one succeeds, the other gets a write conflict and retries. Postgres uses row-level locks (the lock is held in the row's xmax field); WiredTiger uses timestamp-based conflict detection.
BSON and schema flexibility
BSON documents are self-describing — every field has a type tag + length prefix. This means MongoDB never needs a schema to deserialize a document: it reads the type, reads N bytes, moves to the next field. This is why you can store documents with different shapes in the same collection and query any of them — there's no schema to violate at the storage layer.

The cost: BSON is verbose. Every document carries field names as strings. A collection with 10M documents each having a field "last_modified_timestamp" stores that 22-byte string 10M times. Postgres stores the column name once in pg_attribute. For wide documents with many fields, this field-name overhead is significant.
02

Indexes: What Works and What Doesn't

📑
B-Tree + GeoSpatial + Multikey Indexes
MongoDB's index capabilities — and where GIN in Postgres actually beats it
Indexes ▼
Multikey index: indexing array fields
When you index a field that contains an array (tags: ["nosql","database","mongodb"]), MongoDB creates a multikey index — one index entry per array element, all pointing to the same document. A query {tags: "nosql"} hits the B-tree, finds the entry for "nosql", and returns the document. This is semantically similar to Postgres's GIN index on an array column, but implemented differently: MongoDB's is a B-tree with duplicated entries; Postgres GIN maintains a posting list per value.
Why complex nested queries degrade
MongoDB's query planner can use an index on specs.resolution if you've explicitly created one. But it has no statistics on value distribution for nested BSON paths (unlike Postgres's pg_statistic). The planner uses trial sampling (runs candidate plans for 101 documents and picks the fastest) — called plan cache. For stable data patterns this works; for high-cardinality nested fields with skewed distributions, the cached plan can be wrong.
🔑 Critical Distinction
MongoDB's strength is that the entire document is the unit of retrieval — fetch by _id, get everything in one read from the B-tree leaf. Postgres fetching the same entity spread across 5 normalized tables requires 4 JOINs = 4+ index lookups + heap fetches. For "give me this whole entity" access patterns, MongoDB wins on I/O count. For "give me all entities where nested_field = X" access patterns, Postgres's statistics-backed planner often wins on plan quality.
03

Replication + Sharding

🌐
Replica Sets + Oplog
How MongoDB replicates — and why the oplog design matters
Replication ▼
Replica set + oplog
A MongoDB replica set has one primary and N-1 secondaries. The primary writes all changes to its oplog (operations log) — a capped collection stored as a fixed-size ring buffer. Secondaries tail the oplog and apply operations in order. Leader election uses Raft consensus (since MongoDB 3.2).

Oplog is idempotent: each entry is a full document replacement ($set operations are expanded to their final state), not a diff. This makes replication safe to replay but means oplog entries are larger than the original write for small field updates.
Sharding: the shard key is everything
MongoDB sharding partitions data by a shard key. A mongos router hashes or range-partitions the shard key to determine which shard holds a document. A query that includes the shard key is routed to one shard (targeted). A query without the shard key is broadcast to all shards (scatter-gather) and merged by mongos.

The cardinal sin: choosing a low-cardinality shard key (e.g., status: "active"/"inactive"). All "active" documents land on one shard — a hot shard that gets all writes. Same problem as Cassandra's hot partition. The shard key must be high-cardinality, write-distributed, and query-aligned.

What good MongoDB thinking sounds like in an interview

Knows WiredTiger uses document-level MVCC, not the old global lock of MMAPv1
"MongoDB doesn't have transactions" — it has multi-document ACID transactions since v4.0
Explains BSON field-name overhead and why it matters for wide documents at scale
Can't distinguish multikey index from a GIN index; just says "indexes work the same as SQL"
Designs the document shape around the fetch unit: things fetched together live together
Puts everything in one collection with deeply nested paths and expects all queries to be efficient

Graph · Index-Free Adjacency · Property Graph

Neo4j Internals

Neo4j's entire storage model exists to answer one question: how do you traverse relationships in O(1) per hop instead of O(log N) per JOIN? The answer is index-free adjacency — the most important data structure difference between a graph database and a relational one.

01

Index-Free Adjacency: The Core Data Structure

🔗
How Nodes and Relationships Are Stored
The linked-list structure that makes multi-hop traversal O(1) per step
Core ▼
Fixed-size node records
Each node in Neo4j is stored as a fixed-size 15-byte record in a node store file. The record contains: in-use flag, first relationship ID, first property ID, label store ID. Because the record size is fixed, the node with ID n is at byte offset n × 15 in the file — a direct array lookup with no index needed.
Relationship records: the doubly-linked list
Each relationship is a 34-byte record containing: start node ID, end node ID, relationship type, first property ID, and — crucially — four pointers: prev/next relationship for start node and prev/next relationship for end node.

This means every node has a doubly-linked list of all its relationships, stored directly in the relationship store file. To find all neighbors of node N: read node record N (1 disk seek) → follow first_relationship_id → read that relationship record → follow next_relationship_for_N chain. Every step is a fixed-offset file read, not a B-tree lookup.

Why this beats SQL JOINs for deep traversal
In SQL, finding neighbors of user 1 requires: SELECT to_id FROM edges WHERE from_id = 1 → B-tree index lookup on from_id column → heap fetch for each matching row. Cost is O(log N) for the index + O(k) heap fetches where N is total edge count and k is degree.

In Neo4j: node record is at byte id × 15 (direct offset, no index). Relationship chain is followed by pointer arithmetic. Each hop is O(1) — not O(log N). At depth 4 in a social graph: SQL makes 4 index lookups with exponentially growing result sets that must be deduplicated. Neo4j makes 4 pointer follows with result capped by the actual graph structure.
🔑 The Key Insight
The performance of graph traversal in Neo4j is proportional to the subgraph actually traversed, not to the total graph size. Finding friends-of-friends of user 1 in a 1B-node graph takes the same time as in a 1M-node graph, as long as user 1's neighborhood is the same size. SQL's JOIN cost grows with table size, not subgraph size.
02

Property Storage + Cypher Execution

🧩
Property Store + Label Index
How properties and labels are stored, and how Cypher queries enter the graph
Storage ▼
Property storage
Properties (key-value attributes on nodes/relationships) are stored in a separate property store as a linked list of property records, each 41 bytes. Small values (integers, booleans, short strings ≤36 bytes) are stored inline. Larger strings/arrays are stored in a dynamic store (separate file) with a pointer from the property record. The node record points to the first property in its chain.
Label indexes and query entry points
A Cypher query like MATCH (u:User {name: "Alice"}) needs an entry point into the graph. Without an index on :User(name), Neo4j would scan all User nodes — equivalent to a full table scan. With a native B-tree or range index on :User(name), the planner finds Alice's node ID in O(log N) and then traverses from there in O(1) per hop.

This is the query planning analog: graph traversal scales with subgraph size, but the starting point for a traversal still requires an index on node properties if you're not starting from a known node ID.
💬 Interview Signal
Ask: "When does Neo4j require an index, and when is traversal index-free?" Good answer: you need an index to find the entry nodes (those matching label+property predicates). Once you have the entry node IDs, subsequent traversal hops are index-free — pure pointer following. Weak answer: "Neo4j is index-free so it doesn't need indexes" — wrong, it still needs indexes to resolve the initial MATCH predicate.

Explains index-free adjacency: each node has a linked list of relationships, traversal is O(1) per hop via pointer

Knows traversal cost scales with subgraph size, not total graph size

Search · Inverted Index · Lucene Segments

Elasticsearch Internals

Elasticsearch is Lucene distributed. Every performance characteristic — why full-text is fast, why updates are expensive, why aggregations work — traces back to the inverted index and Lucene's immutable segment architecture.

01

The Inverted Index: Why Text Search Is O(1)

📚
Inverted Index + Posting Lists
The data structure that makes "contains laptop" a constant-time lookup
Core ▼
Structure
An inverted index maps every term → sorted list of document IDs that contain it (called a posting list). During indexing, documents are analyzed: text is tokenized, lowercased, stemmed (e.g., "running" → "run"), and stop words removed. Each resulting term is inserted into the index with a pointer to the document.

At query time: "laptop gaming" → lookup "laptop" posting list [doc1, doc3, doc7, ...] + lookup "gaming" posting list [doc3, doc7, doc12, ...] → merge (intersection or union depending on operator) → score results with BM25.
BM25 scoring — what it actually computes
BM25 (Best Match 25) scores each document for a query based on: TF (term frequency — how many times the term appears in the document), IDF (inverse document frequency — how rare the term is across all documents), and document length normalization (penalizes long documents for having the same TF as short ones).

This is what Postgres ts_rank() does NOT have: IDF across the corpus and length normalization. Postgres's ts_rank is purely within-document frequency. A query for "laptop" ranks a 5-word title with one "laptop" the same as a 1000-word review with one "laptop" — wrong. BM25 normalizes by document length.
02

Lucene Segments: Immutable Writes + Merge

🧱
Segment Architecture
Why Elasticsearch is "near real-time" and not instantly consistent
Architecture ▼
Segments and the refresh cycle
Lucene (Elasticsearch's core) builds an inverted index in immutable segments. New documents are buffered in memory (the in-memory buffer). At the refresh interval (default: 1 second), the buffer is flushed to a new segment on disk and becomes searchable. This is what "near real-time" means: up to 1 second latency between indexing and visibility.

Segments are immutable — once written, never modified. A DELETE marks a document as deleted in a bitset (tombstone). An UPDATE is a DELETE + INSERT of a new version. The old version stays in the segment until the segment is merged away.
Segment merging and its implications
As segments accumulate, background merge processes combine small segments into larger ones, physically removing deleted documents (garbage collecting tombstones). During a merge, the merged segments are read sequentially, the new combined segment is written sequentially, and old segments are deleted. This is an LSM-tree pattern applied to inverted indexes.

The implication: heavy indexing + frequent updates = high merge pressure = high I/O. Elasticsearch is not a good primary store for data with high update rates (e.g., order status updating frequently). It's optimized for append-heavy document indexing.

⚠ The Durability Model — Often Missed
The segment flush is NOT the durability point. Elasticsearch maintains a translog (similar to a WAL) — every indexed document is written to the translog before acknowledgment. Segment flushes happen every 30min or when the translog exceeds 512MB. On crash recovery: segments are intact (immutable), and the translog is replayed for the unflushed operations. This is exactly Postgres's WAL + heap file pattern, applied to inverted index segments.

Time-Series · TSM Engine · Columnar Compression

InfluxDB / Time-Series DB Internals

Time-series databases win on compression ratio and aggregation speed because they exploit two properties of metrics data that general-purpose databases cannot: values are monotonically time-ordered, and consecutive values are highly similar. The storage engine is designed entirely around this.

01

TSM Engine: Time-Structured Merge Tree

📦
TSM Files + Columnar Compression
InfluxDB's LSM-tree variant purpose-built for time-ordered float data
Storage ▼
The write path
Writes go to a WAL (for durability) + in-memory Cache. The cache is a map from (measurement, tags, field, timestamp) to value. When the cache reaches threshold, it is flushed to a TSM file (Time-Structured Merge Tree file) — InfluxDB's equivalent of an SSTable.
TSM file layout — columnar within time blocks
A TSM file stores data column-by-column within blocks. For a given series (e.g., cpu,host=web01 usage_idle), all timestamps are stored together as a delta-encoded block, and all values are stored together as a Gorilla-compressed block. This is NOT row storage.

Delta encoding for timestamps: Instead of storing [1609459200, 1609459260, 1609459320, ...], store [1609459200, +60, +60, +60, ...] — then run-length encode the deltas. Regular 60-second intervals compress to near zero.

Gorilla compression for float values: XOR consecutive float64 values. Similar values XOR to small numbers with many leading zeros — stored in 1–2 bytes instead of 8. CPU usage going [72.3, 72.5, 72.2, 73.1] XORs to nearly all zeros. 10-50× compression on typical metrics.
🔑 Why Postgres Row Storage Cannot Compete
Postgres stores each data point as a full row: (OID 4B + MVCC headers 23B + null bitmap + alignment) + timestamp 8B + metric_name text + value 8B = ~60-80 bytes per row. InfluxDB with Gorilla compression stores the same point in ~1-2 bytes for the value + ~1 byte for the timestamp delta. The same 1TB of metrics data = 5-15GB in InfluxDB. This is not a tuning difference — it's a storage model difference.
Continuous queries and rollups
InfluxDB's continuous queries and TimescaleDB's continuous aggregates maintain pre-computed summaries incrementally. When a new data point arrives in time window T, the summary for T is updated — a delta operation, not a full recompute. Postgres's materialized views do a full SELECT ... GROUP BY recompute on every REFRESH. This is why real-time Grafana dashboards on metrics data use time-series DBs — the query hits a pre-computed summary, not billions of raw rows.
02

Series Cardinality: The Hidden Scaling Problem

⚠️
High Cardinality Tags
The #1 operational mistake with time-series databases
Pitfall ▼
What series cardinality means
A "series" in InfluxDB is a unique combination of measurement + tag set. Tags are indexed in an in-memory index (TSI — Time Series Index) for fast series lookup. If you tag a metric with user_id (100M users × 5 metrics × 3 environments = 1.5B series), the in-memory TSI explodes — InfluxDB's performance degrades dramatically because series metadata is kept in RAM.

This is the opposite of what you'd expect: InfluxDB handles billions of data points fine, but millions of unique series (distinct tag combinations) is a known performance ceiling. The rule: tags should be low-cardinality categorical values (host, region, environment). Never use a user_id, request_id, or UUID as a tag.
💬 Interview Signal
Ask: "You want to track API latency per user_id in InfluxDB. What's wrong with this?" The correct answer: user_id as a tag = high-cardinality series explosion. The right approach: store user_id as a field (not a tag), or aggregate to user cohorts and use cohort as the tag. Candidates who don't know this have never operated a time-series DB under load.

Vector · HNSW · Approximate Nearest Neighbor

Vector Database Internals

Vector databases exist because there is no index structure in any traditional database that can perform approximate nearest-neighbor search in high-dimensional space efficiently. The HNSW graph is the key data structure — understanding it explains every performance characteristic.

01

HNSW: The Index That Makes ANN Fast

🕸
Hierarchical Navigable Small World Graph
Why finding 10 similar vectors among 100M is sub-10ms
Core Index ▼
The structure
HNSW is a multi-layer proximity graph. Each vector is a node. Nodes are connected to their nearest neighbors with edges. The graph has multiple layers: the top layer has few nodes with long-range connections (fast coarse navigation), lower layers have more nodes with short-range connections (fine-grained local search), and the bottom layer contains all vectors.

Think of it like a skip list but in high-dimensional space: the top layer is a highway (jump to roughly the right neighborhood), lower layers are local roads (find the exact nearest neighbors).
Query execution
Given query vector Q: start at the entry point in the top layer. Greedily navigate to the neighbor closest to Q at that layer. Descend to the next layer at that node. Repeat — each layer narrows the search to a smaller neighborhood. At the bottom layer, perform a local beam search among the candidates. Return top-K by distance.

Complexity: O(log N) for the hierarchical descent. The critical insight: you never compare Q against all N vectors. You compare against O(log N) nodes during descent plus a small constant of candidates at the bottom layer. For 100M vectors, you might compare against ~2,000 vectors instead of 100M.
The approximate trade-off
HNSW is approximate — it may miss the true nearest neighbor with some probability. The ef_search parameter controls the size of the candidate set during search: higher ef_search = better recall but slower. This is a fundamental trade-off: 95% recall at 5ms vs 99.9% recall at 50ms. For RAG, 95% recall is usually fine. For deduplication or fraud detection, you may need higher recall — which means either higher ef_search or brute force for a small candidate set.
🔑 Why SQL Cannot Do This
The nearest-neighbor problem in 1536 dimensions has no efficient exact solution with a B-tree or any tree-based index — this is the "curse of dimensionality." In high dimensions, all points are approximately equidistant from each other, so tree partitioning (used by k-d trees, ball trees) becomes ineffective past ~20 dimensions. HNSW side-steps this with a graph where proximity is defined by the data distribution itself, not spatial partitioning.
