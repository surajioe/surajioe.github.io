Chapter 01

Three Numbers That Tell a Story

It starts innocently enough. You have a list of URLs. You want to download them. You write a loop. It works. Then someone asks why it's slow, and you fall down a rabbit hole that ends with you understanding exactly how Python schedules work across threads, coroutines, and operating system processes.

Those three benchmark numbers at the top of this page — 2454ms, 1058ms, 736ms — are not just performance figures. They are a window into three fundamentally different models of concurrent execution. Each one has a story, and understanding that story is the difference between writing code that happens to work and code that works for a reason.

Let's start with the slowest.

The Sequential Trap

The single-threaded result of 2454ms comes from code that looks harmless:

# Looks fine. Runs sequentially.
for url in urls:
    downloader.download_files([url])

The problem is what happens inside that loop. Each call to download_files opens a connection, waits for the server to respond, reads the response, and only then returns. While it waits — and for network I/O, waiting is most of the work — your CPU is doing exactly nothing. It's sat there, perfectly capable of issuing a dozen more requests, blocked behind a single one.

The total time is roughly the sum of all individual request times. With three URLs each taking around 800ms, you get 2400ms. This is called I/O-bound sequential execution, and it is the most common performance bug in networked Python code.

The CPU isn't slow. The network isn't slow. The code is slow — because it refuses to do two things at once.

Chapter 02

Enter Threads — And the GIL

The multi-threaded version cuts the time almost in half: 1058ms. The ThreadPoolExecutor fires all three requests roughly simultaneously, and the wall-clock time collapses to approximately the slowest individual request rather than their sum.

with ThreadPoolExecutor(max_workers=10) as executor:
    futures = {url: executor.submit(self.download_file, url) for url in urls}
    for url, future in futures.items():
        download_result[url] = future.result()

This is the correct instinct. But threads carry baggage.

What the GIL Actually Means Here

Python has the Global Interpreter Lock — a mutex that allows only one thread to execute Python bytecode at a time. If you've heard that "threads don't help in Python," this is why. But that statement is misleading for I/O-bound work, and understanding the nuance matters.

When a thread makes a network request, it calls into the OS kernel and releases the GIL while it waits. The OS takes over the socket operation. During that wait, other Python threads are free to execute. So for I/O-bound work, threads genuinely run concurrently — just not on the CPU side.

Scenario GIL impact Threads useful?
Network I/O GIL released during wait Yes — genuine concurrency
File I/O GIL released during syscall Yes — genuine concurrency
Pure Python CPU work GIL held, preempted every ~5ms No — contention overhead
NumPy / C extensions Many release GIL internally Sometimes — depends on library

Why Threads Still Lose to Async

The ~320ms gap between threads (1058ms) and async (736ms) comes from thread overhead. Every OS thread carries an 1–8MB stack, requires OS scheduling decisions, and involves context switches that cost 1–10 microseconds each. With three URLs this is modest. With ten thousand, it becomes catastrophic — you cannot spin up 10,000 OS threads without exhausting memory and grinding the scheduler to a halt.

Threads hit a wall that async does not. That's the story the 736ms result is telling.

Chapter 03

The Event Loop — One Thread to Rule Them All

The async result of 736ms comes from a fundamentally different execution model. Not more threads — fewer. Exactly one.

The asyncio event loop runs on a single thread. It maintains a queue of coroutines that are ready to execute and a registry of things it's waiting on — timers, sockets, futures. On each iteration, it checks what's ready, runs those coroutines until they voluntarily yield, then checks again.

The critical word is voluntarily. Coroutines yield with the await keyword. When you write:

async with session.get(url) as response:
    return await response.read()

Each await is a contract: "I have nothing useful to do right now. Event loop, go run something else. Wake me when my data arrives." The event loop obliges. It switches to another coroutine — at essentially zero cost, because there's no OS involvement, no stack swap, just a function call.

A Common Misconception About asyncio.gather

When you write:

await asyncio.gather(
    self._producer(urls, session),
    self._consumer(results, total),
)

It is tempting to think this spawns two threads. It does not. Both coroutines run on the same thread, in the same event loop, with no GIL interaction whatsoever. They interleave by taking turns at await boundaries:

# What actually happens:
iteration 1: resume _producer → hits await queue.put()  → suspend
iteration 2: resume _consumer → hits await queue.get()  → suspend
iteration 3: resume _producer → fetch completed         → puts item → suspend
iteration 4: resume _consumer → item available          → processes  → suspend
...

The GIL is irrelevant here. There is only one thread — it is never contested.

Because there is only one thread, any synchronous CPU-bound code blocks the entire event loop. A 500ms JSON parse inside an async function freezes every other coroutine for 500ms. The OS cannot preempt it. Only an await can yield control — and CPU work doesn't await anything.

Chapter 04

The Bridge — run_in_executor Explained

So async is ideal for I/O, but breaks down the moment you need CPU work. The solution is a bridge between the async world and a pool of OS processes. That bridge is loop.run_in_executor().

processed_data = await loop.run_in_executor(
    self._process_pool,
    partial(cpu_bound_task, data)
)

This single line does more than it appears. Let's dissect each piece.

Step 1 — partial(cpu_bound_task, data)

The ProcessPoolExecutor sends work to separate OS processes via pickle serialisation. This means the function and its arguments must be picklable. Lambdas are not picklable. Instance methods with non-picklable state are not picklable. functools.partial wrapping a top-level function with plain data is.

This constraint is not obvious and causes real bugs in production.

# FAILS — lambda not picklable
loop.run_in_executor(pool, lambda: cpu_bound_task(data))

# FAILS — instance method with unpicklable state
loop.run_in_executor(pool, self._process)

# WORKS — top-level function + partial
loop.run_in_executor(pool, partial(cpu_bound_task, data))

Step 2 — What the Event Loop Does

Internally, run_in_executor calls pool.submit(fn) which returns a concurrent.futures.Future. It then wraps that into an asyncio.Future. These are different types — the former blocks when you call .result(), the latter suspends a coroutine when you await it.

The await suspends this coroutine and registers a callback: "when the process pool future resolves, set the result on the asyncio future and reschedule this coroutine." The event loop is then free to run everything else.

Event Loop Thread          Worker Process
──────────────────         ──────────────────────
await run_in_executor()
  → submit to pool    ───► pickle(partial(cpu_bound_task, data))
  → suspend coroutine      unpickle in worker process
  → run other work         cpu_bound_task(data) executes
  → run other work         (GIL in THIS process — irrelevant to main)
  → run other work         done → pickle(result)
  ← future resolves   ◄─── send result back via IPC
  → resume coroutine
processed_data = result

Step 3 — The Hidden Cost

Every call to run_in_executor pays a serialisation tax twice — once for the input data travelling to the worker, once for the result travelling back. For large payloads (>10MB) or fast CPU tasks (<10ms), this IPC overhead can exceed the benefit of offloading. It's not a silver bullet — it's a tool for the right situation.

ConditionUse ProcessPoolExecutor?Reason
CPU work > 50msYesPool overhead is amortised
CPU work < 10msNoPickle cost dominates
Result > 10MBCarefulPickle cost scales with size
NumPy / pandasMaybeThey release GIL — ThreadPool may suffice
Pure Python CPUYesFull GIL escape, genuine parallelism
Chapter 05

Backpressure — The Part Everyone Gets Wrong

Here is where most implementations quietly fall apart at scale. The pattern of async I/O feeding a CPU-bound process pool sounds clean in theory. In practice, without deliberate flow control, you have built a time bomb.

What Happens Without a Bounded Queue

Imagine 10,000 URLs. Your async producer is fast — it can fire all 10,000 requests and have results back in memory within seconds. Your process pool has 4 workers and each task takes 500ms. The math is brutal:

# Fetch rate:  ~1000 URLs / second (async, 10 concurrent)
# CPU rate:    4 workers × 2 tasks/sec = 8 tasks / second
# Queue growth: 1000 - 8 = 992 items accumulating per second

# After 10 seconds:
# → 9920 items in memory
# → each at 1MB payload = ~10GB of raw bytes
# → plus pickled copies awaiting IPC
# → plus 9920 asyncio.Futures registered on event loop
# → OOM or catastrophic GC pressure

And it gets worse. ProcessPoolExecutor's internal work queue is a SimpleQueue with no size limit. Calling run_in_executor immediately pickles the data and pushes it into that queue. So you don't just have unprocessed results sitting in memory — you have pickled copies of all of them serialised and waiting. Your memory consumption multiplies.

run_in_executor provides zero backpressure. It accepts work unconditionally. The queue is your responsibility.

The Fix — A Bounded asyncio.Queue

An asyncio.Queue with a maxsize is the correct backpressure mechanism. When the queue is full, await queue.put(item) suspends the producer coroutine. The event loop cannot deliver more items until the consumer pulls one out. This creates natural flow control with zero polling or sleep calls.

# Producer stalls automatically when queue is full
await self._queue.put((url, data))  # ← suspends if maxsize reached

# Consumer frees space, producer resumes
item = await self._queue.get()       # ← frees a slot

Sizing the Queue Correctly

There is a common mistake in queue sizing formulas. The queue does not exist to compensate for a throughput mismatch — you cannot buffer your way out of a rate imbalance. If your fetch rate exceeds your CPU rate, no queue size will save you. The queue's job is to smooth over transient latency spikes.

The correct approach has two steps. First, set your semaphore to cap the fetch rate at or below the CPU throughput:

cpu_workers = 4
fetch_time_ms = 100
cpu_time_ms = 500

# How many concurrent fetches can CPU keep up with?
max_concurrent_fetches = max(1, int(cpu_workers * (fetch_time_ms / cpu_time_ms)))
# = max(1, int(4 × 0.2)) = 1

# Queue only needs to buffer in-flight fetches + small spike room
queue_maxsize = max_concurrent_fetches + cpu_workers
# = 1 + 4 = 5

semaphore = asyncio.Semaphore(max_concurrent_fetches)  # rate control
queue = asyncio.Queue(maxsize=queue_maxsize)           # spike buffer

With this in place, memory consumption becomes constant regardless of how many URLs you process. The system self-regulates rather than accumulating unboundedly.

The formula cpu_workers × (fetch_time / cpu_time) gives the correct value for max_concurrent_fetches — not for queue_maxsize. It controls how many fetches can run simultaneously while keeping CPU workers fully fed without overflow. This is a semaphore value, not a queue size.

Chapter 06

The Full Architecture — Three Gates, One Pipeline

When you put all the pieces together, the system has three distinct control points, each governing a different resource:

URLs
  │
  ▼
[Semaphore — controls network connections]
  max_concurrent_fetches = cpu_workers × (fetch_time / cpu_time)
  │
  ▼ (raw bytes)
[asyncio.Queue — controls memory / backpressure]
  maxsize = max_concurrent_fetches + cpu_workers
  │  producer blocks here when full
  ▼
[ProcessPoolExecutor — controls CPU core utilisation]
  cpu_workers = number of physical cores
  internal _work_queue is unbounded — never reaches it at volume
  because asyncio.Queue gates all input
  │
  ▼
final results dict

Each gate is necessary. Remove the semaphore and your network stack gets hammered. Remove the queue size limit and memory explodes before CPU work starts. Remove the process pool and CPU work blocks the event loop.

The Complete Implementation

import asyncio
import aiohttp
from concurrent.futures import ProcessPoolExecutor
from functools import partial

# Top-level: required for pickle across process boundary
def cpu_bound_task(data: bytes) -> dict:
    import json
    return json.loads(data)

class AsyncFileDownloaderWithProcessing:
    def __init__(self, cpu_workers=4, fetch_time_ms=100, cpu_time_ms=500):
        max_fetches = max(1, int(cpu_workers * (fetch_time_ms / cpu_time_ms)))
        self._semaphore = asyncio.Semaphore(max_fetches)
        self._queue = asyncio.Queue(maxsize=max_fetches + cpu_workers)
        self._pool = ProcessPoolExecutor(max_workers=cpu_workers)

    async def _fetch(self, url, session):
        async with self._semaphore:
            async with session.get(url) as r:
                return url, await r.read() if r.status == 200 else Exception(r.status)

    async def _producer(self, urls, session):
        for coro in asyncio.as_completed([self._fetch(u, session) for u in urls]):
            await self._queue.put(await coro)  # stalls if queue full
        await self._queue.put(None)             # sentinel

    async def _consumer(self, results, total):
        loop = asyncio.get_running_loop()
        processed = 0
        while processed < total:
            item = await self._queue.get()
            if item is None: break
            url, data = item
            if isinstance(data, Exception):
                results[url] = data
            else:
                # yield to event loop; resume when process finishes
                results[url] = await loop.run_in_executor(
                    self._pool, partial(cpu_bound_task, data)
                )
            processed += 1

    async def __aenter__(self): return self

    async def __aexit__(self, *args):
        self._pool.shutdown(wait=True)  # clean process teardown
Chapter 07

The Bugs That Hide in Plain Sight

The original code that started this journey had several subtle bugs worth examining — not because they caused obvious crashes, but because they degrade silently under load.

Bug 1 — Semaphore Inside the Retry Loop

for attempt in range(self._max_retries):
    async with self._semaphore:         # re-acquired each retry
        async with session.get(url) as response:
            if response.status != 200:
                ex = Exception(response)
                # falls through — retries immediately, no backoff

A non-200 response sets ex and falls through to the next loop iteration without any await between retries. This means all retry attempts execute as fast as the CPU can run, re-acquiring the semaphore on each attempt. Under high failure rates, this starves other coroutines waiting for a semaphore slot. Add exponential backoff with await asyncio.sleep(2 ** attempt) between retries.

Bug 2 — ClientTimeout Applied Twice

The AsyncFileDownloader creates a ClientSession with a session-level timeout, then also passes timeout=self._timeout to each individual session.get() call. The per-request timeout overrides the session timeout, making the session-level setting meaningless. More critically, ClientTimeout is used but never imported — this raises a NameError at runtime, not a graceful timeout.

Bug 3 — return_exceptions=False in gather

results = await asyncio.gather(*tasks, return_exceptions=False)

With return_exceptions=False (the default), a single exception from any task cancels all remaining tasks and propagates immediately. For a URL downloader, one bad URL aborts all in-flight downloads. The correct approach is return_exceptions=True, which returns exceptions as values in the results list for you to handle per-URL.

Bug 4 — Process Pool Lifecycle

If the ProcessPoolExecutor is shut down before all coroutines have finished — through an unhandled exception, early return, or missing context manager — any pending await loop.run_in_executor() calls raise RuntimeError: cannot schedule new futures after shutdown. This exception doesn't surface through normal aiohttp error handling. Always tie the pool lifecycle to an async with context manager.

Chapter 08

Scaling Beyond One Process — The Full Picture

Everything so far operates within a single Python process. That is appropriate for I/O-bound work with moderate CPU processing. But there are cases where you need to saturate multiple CPU cores with both I/O and CPU work simultaneously. The architecture changes.

A single event loop cannot use multiple CPU cores. This is a fundamental constraint. To scale across cores, you need multiple processes — each running their own event loop. multiprocessing handles the cross-process coordination:

from multiprocessing import Pool
import asyncio

def worker(urls_chunk):
    # Each process gets its own event loop
    async def run():
        async with AsyncFileDownloaderWithProcessing() as d:
            return await d.download_and_process(urls_chunk)
    return asyncio.run(run())

# Split URLs across CPU cores
with Pool(processes=os.cpu_count()) as pool:
    chunks = chunk(urls, size=len(urls) // os.cpu_count())
    all_results = pool.map(worker, chunks)

This gives you async concurrency within each process and true CPU parallelism across processes. ThreadPoolExecutor alone cannot achieve this — it is limited by the GIL for CPU work and limited by OS thread caps for I/O work. It sits in an awkward middle ground at extreme scale.

ScaleRecommended approach
< 100 URLsAsync with aiohttp — simplest, fast enough
100 – 10k URLsAsync + ProcessPoolExecutor for CPU work, bounded queue
> 10k URLsMultiple processes, each with async event loop + process pool
100k+ URLsDistributed system (Celery, Ray, or equivalent)

The async vs threads debate is often framed as a competition. It isn't. Async handles I/O concurrency with minimal resources. Threads provide OS-level preemption for mixed workloads. Processes escape the GIL for CPU parallelism. The real skill is knowing which tool governs which resource — and building systems where each resource has exactly one control point that can be reasoned about independently.
