TCP — Transmission Control Protocol

Professor's Lecture

TCP is a contract between two endpoints. When you send a byte, TCP guarantees it arrives, it arrives exactly once, and it arrives in order. To enforce that contract it pays a tax: the three-way handshake, sequence numbers, acknowledgments, retransmission timers, and congestion control. Every higher-level protocol in this guide is either running on top of TCP or was specifically designed to escape one of TCP's penalties.

The three-way handshake costs one round-trip before the first byte of application data. TLS adds another 1–2 RTTs. On a 100ms transatlantic link that's 300ms of latency before your HTTP GET even begins. This is why CDNs, connection pooling, and HTTP/2 multiplexing exist — they amortize this cost across many requests.

Congestion control (CUBIC on Linux, BBR at Google) is the other critical concept. TCP probes network capacity by increasing its congestion window until it detects packet loss, then backs off. This means a brand-new connection starts slow — it literally has a slow-start phase. A connection that has been warm for 10 seconds will have a much larger window and much higher throughput than one that just opened.

Key TCP Mechanics You Must Know

Flow Control vs Congestion Control

Flow control is receiver-driven: the receiver advertises a receive window (rwnd) to tell the sender how much buffer space it has. Congestion control is network-driven: the sender maintains a congestion window (cwnd) based on inferred network conditions. The actual send rate is min(rwnd, cwnd).

TCP Options that matter in system design

  • SO_REUSEADDR / SO_REUSEPORT — lets multiple sockets bind to the same port; critical for zero-downtime restarts and multi-process servers
  • TCP_NODELAY — disables Nagle's algorithm; critical for latency-sensitive apps (Redis uses this)
  • TCP_KEEPALIVE — detects dead connections; but it's slow (2h default). Use application-level heartbeats instead
  • SO_LINGER — controls behavior on close(); affects TIME_WAIT state

TIME_WAIT — the production gotcha

After closing a connection, the OS keeps it in TIME_WAIT for 2×MSL (~60s). Under high request rates, you can exhaust ephemeral ports (65k). Solutions: net.ipv4.tcp_tw_reuse=1, or better: connection pooling to reduce close() frequency.

import socket
import struct
import threading
import time

# --- Length-prefixed framing (CRITICAL: raw TCP is a BYTE STREAM, not message stream) ---
# Without framing, you'll read partial messages. This is the #1 TCP beginner mistake.

def send_message(sock, msg: bytes):
    """Prefix every message with its 4-byte length."""
    length = struct.pack('!I', len(msg))  # network byte order
    sock.sendall(length + msg)

def recv_message(sock) -> bytes:
    """Read exactly 4 bytes for length, then read that many bytes."""
    raw_len = _recv_exactly(sock, 4)
    if not raw_len:
        raise ConnectionResetError("Connection closed")
    msg_len = struct.unpack('!I', raw_len)[0]
    return _recv_exactly(sock, msg_len)

def _recv_exactly(sock, n: int) -> bytes:
    """Loop until n bytes received — TCP may deliver in fragments."""
    buf = b''
    while len(buf) < n:
        chunk = sock.recv(n - len(buf))
        if not chunk:
            return None
        buf += chunk
    return buf

# --- Server with SO_REUSEADDR + TCP_NODELAY + connection-level keep-alive ---

class TCPServer:
    def __init__(self, host='0.0.0.0', port=9000):
        self.sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)

        # SO_REUSEADDR: allows bind() immediately after restart without waiting
        # for TIME_WAIT to expire. Essential for zero-downtime deploys.
        self.sock.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)

        # SO_REUSEPORT: allows multiple processes to bind to same port.
        # The kernel load-balances incoming connections. Use with pre-fork servers.
        self.sock.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEPORT, 1)

        self.sock.bind((host, port))
        self.sock.listen(1024)  # backlog: queue depth for unaccepted connections

    def handle_client(self, conn, addr):
        # TCP_NODELAY: disable Nagle's algorithm.
        # Nagle buffers small writes to reduce packet count — terrible for latency.
        # Always set this for request/response workloads.
        conn.setsockopt(socket.IPPROTO_TCP, socket.TCP_NODELAY, 1)

        # TCP keepalive at OS level — detects half-open connections
        # idle=10s, interval=5s, count=3 probes before declaring dead
        conn.setsockopt(socket.SOL_SOCKET, socket.SO_KEEPALIVE, 1)
        conn.setsockopt(socket.IPPROTO_TCP, socket.TCP_KEEPIDLE, 10)
        conn.setsockopt(socket.IPPROTO_TCP, socket.TCP_KEEPINTVL, 5)
        conn.setsockopt(socket.IPPROTO_TCP, socket.TCP_KEEPCNT, 3)

        try:
            while True:
                msg = recv_message(conn)
                # Echo server — replace with your logic
                send_message(conn, b'ACK: ' + msg)
        except (ConnectionResetError, BrokenPipeError):
            pass
        finally:
            conn.close()

    def serve_forever(self):
        while True:
            conn, addr = self.sock.accept()
            t = threading.Thread(target=self.handle_client, args=(conn, addr), daemon=True)
            t.start()

TCP Head-of-Line Blocking — The Deep Cut

Professor's Lecture

TCP's ordering guarantee is implemented in the kernel's receive buffer using a concept called the reorder buffer (or out-of-order queue). When segments arrive, the kernel places them in sequence-number order. The critical rule: the kernel will not deliver any bytes to the application until there are no gaps in the sequence space from the last delivered byte.

So if segments with sequence numbers 1–1000, 2001–3000, and 3001–4000 have all arrived, but segment 1001–2000 is missing, the kernel holds all three arrived segments and delivers nothing. The application's recv() call blocks. This is TCP head-of-line blocking — one lost segment freezes the entire connection's data delivery, no matter how many other segments have arrived safely.

Now layer HTTP/2 on top. HTTP/2 multiplexes independent streams — stream 1 (an image), stream 3 (a CSS file), stream 5 (an API call) — over a single TCP connection. From HTTP/2's perspective these are completely unrelated. But TCP has no concept of streams. It sees one byte stream. Drop a TCP segment that belongs to stream 3's data, and TCP freezes delivery of all bytes — including perfectly intact data for streams 1 and 5. HTTP/2 cannot unblock them. The multiplexing that HTTP/2 provides at the application layer is entirely undone by TCP's ordering requirement at the transport layer.

This is not a bug in TCP — it is a direct consequence of its correctness guarantee. You cannot have ordered reliable delivery without blocking on gaps. QUIC's answer is to implement reliability per stream at the transport level: loss in stream 3 only blocks stream 3's delivery; streams 1 and 5 continue unaffected.

  Each QUIC stream has its own delivery cursor.
  Loss in one stream does NOT affect others.

Why can't TCP fix this with a protocol extension?

The short answer is ossification. Middleboxes — NATs, firewalls, performance-enhancing proxies, WAN optimizers — have TCP's semantics baked into their silicon and firmware. Any TCP extension that changes the ordering semantics would be silently dropped, rewritten, or misinterpreted by the billions of middleboxes deployed in the field. RFC 8312 (CUBIC) and RFC 7413 (TCP Fast Open) both suffered from middlebox interference. This is exactly why QUIC runs over UDP — UDP packets are opaque to middleboxes, so QUIC's per-stream ordering can be implemented freely in userspace without hardware interference.

When does TCP HOL blocking actually hurt you in production? On a datacenter network (0.001% packet loss, sub-millisecond RTT), TCP HOL blocking is negligible — retransmits happen in microseconds. The pain is on mobile networks (1–3% loss, 50–200ms RTT) and satellite links (0.5–2% loss, 600ms RTT). At 1% loss with HTTP/2, roughly 1 in 100 connections experiences a freeze measurable in tens of milliseconds. At the 99th percentile tail latency this compounds badly. Google's QUIC rollout showed 30% p99 latency improvement on mobile, nearly all attributable to eliminating transport-level HOL blocking.

import socket
import threading
import time
import struct

# ─── Demonstrate TCP HOL blocking ────────────────────────────────────────────
# We simulate it by writing to the TCP receive buffer out of order using
# two separate connections and showing that a stall on one blocks the reader.
#
# In a real kernel, you cannot inject out-of-order segments from Python
# (that's below the socket API). This demo instead shows the EFFECT:
# a slow sender on one "stream" (simulated by a thread) blocks a shared reader.

def tcp_hol_simulation():
    """
    Simulate two logical streams over ONE TCP connection.
    Stream A sends fast. Stream B introduces a 200ms gap (simulating loss + retransmit).
    Observation: Stream A's data is stuck in the app even though it arrived fine —
    because we must read the connection serially.
    """
    server_sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    server_sock.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
    server_sock.bind(('127.0.0.1', 19876))
    server_sock.listen(1)

    def server():
        conn, _ = server_sock.accept()
        conn.setsockopt(socket.IPPROTO_TCP, socket.TCP_NODELAY, 1)
        events = []
        start = time.monotonic()

        # Read all messages in arrival order — no choice, TCP is serial
        for _ in range(6):
            raw = conn.recv(64)
            t = time.monotonic() - start
            events.append((t, raw.decode().strip()))

        for ts, msg in events:
            print(f"  [{ts:.3f}s] received: {msg}")
        conn.close()
        server_sock.close()

    threading.Thread(target=server, daemon=True).start()
    time.sleep(0.05)

    client = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    client.connect(('127.0.0.1', 19876))
    client.setsockopt(socket.IPPROTO_TCP, socket.TCP_NODELAY, 1)

    def stream_a():
        # Stream A: sends promptly at t=0, t=0.05, t=0.1
        for i in range(3):
            client.sendall(f"A{i} (no delay)   \n".encode())
            time.sleep(0.05)

    def stream_b():
        # Stream B: first message fine, then simulates a 200ms "loss+retransmit"
        client.sendall(b"B0 (on time)      \n")
        time.sleep(0.2)   # <-- simulated retransmit delay
        client.sendall(b"B1 (200ms late)   \n")
        time.sleep(0.05)
        client.sendall(b"B2 (on time)      \n")

    # In real HTTP/2 over TCP, A and B frames are interleaved on ONE connection.
    # The reader cannot skip B's stall to deliver A's waiting data.
    ta = threading.Thread(target=stream_a)
    tb = threading.Thread(target=stream_b)
    ta.start(); tb.start()
    ta.join(); tb.join()
    client.close()

# ─── Contrast: UDP allows out-of-order delivery ──────────────────────────────
# With UDP datagrams, each packet is independent. The receiver can process
# whatever arrives — no blocking on gaps. QUIC builds on this property.

def udp_no_hol_demo():
    """
    UDP receiver processes datagrams as they arrive.
    A delayed datagram from 'stream B' does NOT block 'stream A' datagrams.
    QUIC builds per-stream ordering on top of exactly this property.
    """
    server_sock = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
    server_sock.bind(('127.0.0.1', 19877))
    server_sock.settimeout(1.0)

    def udp_server():
        start = time.monotonic()
        received = []
        while True:
            try:
                data, _ = server_sock.recvfrom(64)
                t = time.monotonic() - start
                received.append((t, data.decode().strip()))
            except socket.timeout:
                break
        for ts, msg in received:
            print(f"  [{ts:.3f}s] datagram: {msg}")
        server_sock.close()

    threading.Thread(target=udp_server, daemon=True).start()
    time.sleep(0.05)

    client = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
    target = ('127.0.0.1', 19877)

    # Stream A datagrams: arrive promptly
    for i in range(3):
        client.sendto(f"A{i} stream_a     ".encode(), target)
        time.sleep(0.05)

    # Stream B: B0 on time, B1 delayed 200ms (simulating loss), B2 on time
    client.sendto(b"B0 stream_b       ", target)
    time.sleep(0.2)   # B1 arrives late — but A's datagrams already delivered
    client.sendto(b"B1 late(200ms)    ", target)
    client.sendto(b"B2 stream_b       ", target)
    client.close()

Deep-Dive Questions
Explain exactly why a dropped TCP segment causes HOL blocking across unrelated HTTP/2 streams.
TCP is a byte-stream protocol with a single sequence space shared across the entire connection. The kernel's receive buffer tracks one "next expected sequence number." When a segment is missing, the kernel holds all subsequent arrived segments in an out-of-order queue and will not deliver any of them to the application — regardless of their content — until the gap is filled. HTTP/2 multiplexes streams by interleaving frames on that byte stream, but TCP has no awareness of stream boundaries. Drop one TCP segment carrying stream 3 frames, and the kernel withholds all data including fully-intact frames for streams 1 and 5. The HTTP/2 layer never gets a chance to deliver them. The application experiences a stall on all active streams until the retransmit completes — typically one RTT, but up to several RTTs under loss + congestion.
Why can't TCP solve its own HOL blocking problem the way QUIC does, even with a new TCP extension?
Two reasons. First, semantic impossibility: TCP's contract is ordered reliable delivery of a byte stream. To support per-stream unblocking, TCP would need to know which bytes belong to which stream — that's application-layer semantics. Adding it to TCP conflates transport and application concerns and would require every TCP implementation (in every OS kernel) to be updated. Second, ossification: the deployed base of middleboxes — NAT devices, firewalls, WAN optimisers, DPI appliances — have TCP semantics hardcoded. TCP extensions that deviate from expected behavior are silently dropped or corrupted. TCP Fast Open was specified in 2014 and still sees <5% deployment because middleboxes block SYN data. Any "streams-aware TCP" extension would face the same fate. QUIC sidesteps this entirely: it runs over UDP (treated as opaque by middleboxes) and implements its own reliability per-stream in userspace, where no middlebox can interfere.
Deep-Dive Questions
Explain what happens to TCP throughput on a 100ms RTT link with a 64KB receive window.
Throughput is bounded by window_size / RTT = 65536 bytes / 0.1s ≈ 5.2 Mbps. This is the bandwidth-delay product problem. On high-latency links you must tune net.core.rmem_max and net.ipv4.tcp_rmem to increase the window, or use HTTP/2 which has its own flow control per stream independent of the OS TCP window.
When would you choose UDP over TCP in a backend system?
When the application can tolerate or handle loss itself and latency dominates: DNS (single packet, app retries), real-time gaming (stale frames useless), video conferencing (FEC handles loss), telemetry/metrics (occasional loss acceptable). QUIC is built on UDP because it wants to implement its own reliability per-stream, escaping TCP's head-of-line blocking where loss in stream A stalls stream B.
You're seeing high TIME_WAIT connections on a service making many short-lived outbound connections. How do you fix it?
Three approaches, in preference order: (1) Use connection pooling — stop closing connections between requests. (2) Enable net.ipv4.tcp_tw_reuse=1 — allows reusing TIME_WAIT sockets for outbound connections when safe (requires timestamps). (3) If the server is the closer, use SO_LINGER with l_onoff=1, l_linger=0 to send RST instead of FIN/FIN-ACK, bypassing TIME_WAIT entirely — but this can cause data loss if the peer hasn't read all data.

HTTP/1.1 — The Workhorse

Professor's Lecture

HTTP/1.1 added persistent connections (keep-alive) over HTTP/1.0's close-after-every-response. This was transformative — it meant you paid the TCP handshake cost once per connection rather than once per request. But HTTP/1.1 is fundamentally serial: while one request/response cycle is in flight, no other request can use that connection. Browsers work around this by opening 6 connections per origin — a hack that wastes sockets and causes thundering-herd on the server.

HTTP pipelining was supposed to fix this — send multiple requests without waiting for responses. It was specified in 1999 and virtually never deployed in production because proxy servers couldn't handle it correctly. HTTP/2 replaced pipelining with proper multiplexing.

Connection keep-alive lifecycle

WITHOUT keep-alive (HTTP/1.0):
  TCP handshake → Request → Response → TCP close   (repeats every request)
  Cost: 1 RTT per request just for TCP

WITH keep-alive (HTTP/1.1 default):
  TCP handshake → Req1 → Resp1 → Req2 → Resp2 → Req3 → Resp3 → [timeout/close]
  Cost: 1 RTT amortized across N requests
  But: Req2 must WAIT for Resp1 — head-of-line blocking at HTTP layer

import httpx
import asyncio
import time
from dataclasses import dataclass, field
from enum import Enum, auto
from typing import Optional

# ─── Circuit Breaker ───────────────────────────────────────────────────────────
# Prevents cascading failures. If downstream is unhealthy, fail fast instead of
# queuing threads/connections that will also fail.

class CircuitState(Enum):
    CLOSED = auto()    # Normal operation
    OPEN = auto()      # Fail fast, no requests forwarded
    HALF_OPEN = auto() # Probe: allow one request through

@dataclass
class CircuitBreaker:
    failure_threshold: int = 5
    recovery_timeout: float = 30.0  # seconds before OPEN → HALF_OPEN
    _state: CircuitState = field(default=CircuitState.CLOSED, init=False)
    _failure_count: int = field(default=0, init=False)
    _opened_at: Optional[float] = field(default=None, init=False)

    def is_open(self) -> bool:
        if self._state == CircuitState.OPEN:
            if time.monotonic() - self._opened_at > self.recovery_timeout:
                self._state = CircuitState.HALF_OPEN
                return False
            return True
        return False

    def record_success(self):
        self._failure_count = 0
        self._state = CircuitState.CLOSED

    def record_failure(self):
        self._failure_count += 1
        if self._failure_count >= self.failure_threshold:
            self._state = CircuitState.OPEN
            self._opened_at = time.monotonic()

# ─── Resilient HTTP Client ────────────────────────────────────────────────────

class ResilientHTTPClient:
    """
    Production-grade HTTP client with:
    - Connection pooling (shared across requests, not created per call)
    - Exponential backoff with jitter
    - Circuit breaker
    - Timeout budget (connect timeout separate from read timeout)
    """
    def __init__(self, base_url: str, max_retries=3):
        self.base_url = base_url
        self.max_retries = max_retries
        self.circuit = CircuitBreaker()

        # httpx.AsyncClient maintains a connection pool.
        # limits controls pool size — tune based on your concurrency needs.
        # CRITICAL: Instantiate once, reuse across requests. Never create per-request.
        self._client = httpx.AsyncClient(
            base_url=base_url,
            limits=httpx.Limits(
                max_connections=100,       # total connections
                max_keepalive_connections=20, # warm idle connections
                keepalive_expiry=30.0,       # evict after 30s idle
            ),
            timeout=httpx.Timeout(
                connect=2.0,  # fail fast on connection, not data
                read=10.0,    # reading response body
                write=5.0,
                pool=1.0,    # time waiting for a connection from pool
            ),
            http2=True,  # prefer HTTP/2 if server supports it
        )

    async def get(self, path: str) -> httpx.Response:
        if self.circuit.is_open():
            raise RuntimeError("Circuit OPEN: downstream service is unhealthy")

        last_exc = None
        for attempt in range(self.max_retries):
            try:
                resp = await self._client.get(path)
                resp.raise_for_status()
                self.circuit.record_success()
                return resp

            except (httpx.ConnectError, httpx.TimeoutException, httpx.HTTPStatusError) as e:
                self.circuit.record_failure()
                last_exc = e
                # Exponential backoff with full jitter: prevents thundering herd.
                # cap at 10s, add random jitter so all retries don't fire simultaneously
                wait = min(10.0, (2 ** attempt) * 0.5)
                import random
                wait += random.uniform(0, wait)  # full jitter
                await asyncio.sleep(wait)

        raise last_exc

    async def close(self):
        await self._client.aclose()
Reliability Trap Retry on HTTP 500 (server error) but NOT on HTTP 400 (client error — retrying is pointless). Also don't retry non-idempotent requests (POST) without idempotency keys. The Stripe and Twilio APIs demonstrate this pattern: client generates a UUID, sends it in the request header, and the server deduplicates.
Deep-Dive Questions
Why is exponential backoff with jitter better than exponential backoff without jitter?
Without jitter, if N clients all hit an error at time T, they all back off for the same interval and retry simultaneously at T+2s, T+4s, etc. — the thundering herd just happens in waves. Jitter spreads retries uniformly across the backoff window. AWS's "Exponential Backoff and Jitter" blog (2015) showed "full jitter" outperforms decorrelated jitter and plain exponential backoff under contention.
How does a circuit breaker differ from a timeout? Why do you need both?
A timeout is per-request protection: it bounds how long any single call can block. A circuit breaker is system-level protection: after enough timeouts/errors, it stops sending requests entirely, giving the downstream time to recover and preventing thread/connection pool exhaustion in your service. Without a circuit breaker, a slow downstream can cause every thread in your pool to block at the timeout limit simultaneously, making your service appear dead even though the problem is downstream.

HTTP/2 — Binary Multiplexing

Professor's Lecture

HTTP/2 (RFC 7540, 2015) solves HTTP/1.1's head-of-line blocking at the application layer by introducing streams — independent, multiplexed request/response pairs over a single TCP connection. Each stream gets a stream ID, and frames from different streams are interleaved freely. A slow response to request #1 doesn't block request #2.

HPACK header compression is the other major win. HTTP headers are repetitive — User-Agent, Accept, Authorization are sent identically on every request. HPACK maintains a shared compression table between client and server, so repeated headers are replaced with a 1-byte index. This can reduce header overhead by 85–88% in practice.

The critical caveat: HTTP/2 only eliminates head-of-line blocking at the HTTP layer. At the TCP layer, a single lost packet still blocks all streams until retransmission. This is exactly the problem HTTP/3 addresses.

HTTP/1.1: 6 connections × serial requests
  conn1: [Req A ——— Resp A]
  conn2: [Req B ——— Resp B]
  conn3: [Req C ——————————— Resp C (slow)]   ← wastes connection
  conn4: [Req D — Resp D]
  conn5: [Req E — Resp E]
  conn6: [Req F — Resp F]

HTTP/2: 1 connection × interleaved streams
  stream 1: [Req A frames] ... [Resp A frames]
  stream 3: [Req B frames]     [Resp B frames]
  stream 5: [Req C frames]              [Resp C (slow, doesn't block others)]
  stream 7: [Req D frames] [Resp D]

Deep-Dive Questions
HTTP/2 is supposed to eliminate the need for domain sharding. Why might you keep it anyway?
Domain sharding (serving assets from static1.example.com, static2.example.com) bypasses browser per-origin connection limits for HTTP/1.1. HTTP/2 makes this unnecessary because multiplexing handles concurrency on one connection. The argument to sharding goes away; the argument to separate origins does not.
Explain HPACK's security vulnerability and how it was mitigated.
CRIME (2012) and BREACH attacks showed that if an attacker can inject partial plaintext (e.g., CSRF token guesses) into a compressed stream and observe the compressed size, they can deduce the secret byte-by-byte because compression ratio reveals shared substrings. HPACK mitigates this with the Huffman-only compression mode for sensitive headers and by never compressing headers that the client controls alongside server secrets in the same compression context. TLS also helps by separating compression from encryption (unlike SSL-era DEFLATE).

HTTP/3 & QUIC — Eliminating Transport HOL Blocking

Professor's Lecture

HTTP/2 moved head-of-line blocking from the application layer to the transport layer. On a lossy network — mobile, satellite, congested WiFi — a single lost TCP segment blocks all HTTP/2 streams until that segment is retransmitted. The solution is radical: don't use TCP.

QUIC (RFC 9000) reimplements reliable ordered delivery per stream on top of UDP. Loss in stream A only affects stream A. QUIC also merges the transport and TLS handshakes: a new connection requires one RTT (zero for session resumption with 0-RTT), versus TCP's minimum of two RTTs (SYN+ACK, then TLS). Connection migration is another killer feature: when your phone switches from WiFi to 4G, the IP address changes. TCP breaks (the 4-tuple changes). QUIC uses a connection ID independent of the IP:port pair, so the connection survives the network change.

QUIC vs TCP+TLS comparison:

TCP + TLS 1.3:
  SYN → SYN-ACK → ACK                  (1 RTT: TCP handshake)
  ClientHello → ServerHello → Finished  (1 RTT: TLS 1.3)
  ─────────────────────────────────────
  First data: 2 RTTs minimum

QUIC (new connection):
  Initial → Handshake → 1-RTT           (1 RTT: transport + TLS merged)
  ─────────────────────────────────────
  First data: 1 RTT

QUIC (session resumption, 0-RTT):
  Initial + Early Data (sent immediately, before handshake completes)
  ─────────────────────────────────────
  First data: 0 RTTs
  ⚠ 0-RTT is replay-vulnerable — only use for idempotent requests

Packet loss scenario (3 streams):
  TCP:   stream1 ✓ stream2 ✓ [LOSS] stream3 ✓ stream1 ✓ [WAIT FOR RETRANSMIT] ...
         Everything stalls.
  QUIC:  stream1 ✓ [stream2 stalls] stream3 ✓ stream1 ✓ [stream2 retransmits]
         Only the affected stream stalls.
When HTTP/3 matters most Real-world gains are highest on lossy networks (mobile, 1–2% packet loss). On a datacenter network with 0.001% loss, HTTP/2 and HTTP/3 are essentially equivalent. Google reported ~3% median latency improvement and 30% improvement at the 99th percentile for mobile users on lossy networks.
Deep-Dive Questions
Why is QUIC built on UDP? Doesn't that mean it has to re-implement TCP's reliability?
Yes, that's exactly right — QUIC reimplements reliability, congestion control (NewReno, CUBIC, BBR), and flow control on top of UDP. The reason for UDP is ossification: middleboxes (firewalls, NATs, load balancers) have hardcoded TCP behavior and would corrupt or block any new TCP extensions. UDP is treated as opaque payload and passes through. QUIC's "complexity tax" is worth paying because the gains — per-stream loss recovery, 0-RTT handshake, connection migration — require protocol-level changes that TCP's ossification makes impossible to deploy.
What are the operational challenges of deploying HTTP/3 in your infrastructure?
Several real ones: (1) UDP is often rate-limited or blocked by enterprise firewalls — you need graceful fallback to HTTP/2. (2) Load balancers need to understand QUIC's connection IDs to route consistently, since IP:port changes with connection migration. (3) QUIC is CPU-intensive because encryption is mandatory and in userspace, unlike TCP which offloads to the kernel. (4) Observability tooling (packet captures, Wireshark) works differently with QUIC's encrypted headers. (5) Server-side implementation maturity — nginx QUIC support is still relatively new vs. its battle-tested HTTP/2 stack.

WebSocket — Full-Duplex Persistent Channels

Professor's Lecture

WebSocket solves a fundamental HTTP limitation: HTTP is client-initiated. The server cannot send data to the client without the client asking first. Long-polling was the hack — the client sends a request, the server holds it open until it has data to send, then the client immediately sends another. This works but wastes resources and has high latency.

WebSocket upgrades an existing HTTP/1.1 connection to a persistent, full-duplex channel. The upgrade is an HTTP handshake — which means it traverses HTTP infrastructure (proxies, load balancers) gracefully. Once upgraded, frames can flow in both directions independently, with minimal framing overhead (2–10 bytes vs. hundreds of bytes for HTTP headers).

The operational reality: WebSocket connections are long-lived, which means your servers hold connection state. This is the central scalability challenge. If a user's WebSocket is connected to server A and server A goes down, the client must reconnect — possibly to server B, which has no knowledge of the prior session. You must externalize connection state to a shared store (Redis Pub/Sub is the canonical pattern) or accept that reconnection is part of the protocol.

WebSocket Upgrade Handshake:
  Client → Server:
    GET /ws HTTP/1.1
    Upgrade: websocket
    Connection: Upgrade
    Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==   ← random base64
    Sec-WebSocket-Version: 13

  Server → Client:
    HTTP/1.1 101 Switching Protocols
    Upgrade: websocket
    Connection: Upgrade
    Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=   ← SHA1(key + GUID)

  After this point: raw WebSocket frames, no HTTP overhead.

WebSocket Frame Structure (minimal overhead):
  ┌─────────────────────────────────────┐
  │ FIN(1) RSV(3) Opcode(4) MASK(1) Payload Length(7/16/64 bits) │
  │ Masking Key (32 bits, client→server only)                     │
  │ Payload Data                                                   │
  └─────────────────────────────────────┘
  Overhead: 2–10 bytes vs ~500 bytes for HTTP headers

from fastapi import FastAPI, WebSocket, WebSocketDisconnect
import redis.asyncio as aioredis
import asyncio
import json

# ─── Connection Manager ───────────────────────────────────────────────────────
# Manages all WebSocket connections on THIS server instance.
# Cross-server fanout is handled by Redis Pub/Sub.

class ConnectionManager:
    def __init__(self):
        # room_id → set of WebSocket connections on this server
        self.rooms: dict[str, set[WebSocket]] = {}

    async def join(self, room_id: str, ws: WebSocket):
        await ws.accept()
        self.rooms.setdefault(room_id, set()).add(ws)

    def leave(self, room_id: str, ws: WebSocket):
        if room_id in self.rooms:
            self.rooms[room_id].discard(ws)

    async def broadcast_local(self, room_id: str, msg: str, exclude: WebSocket = None):
        """Send to all connections on this server for the given room."""
        dead = set()
        for ws in self.rooms.get(room_id, set()):
            if ws == exclude: continue
            try:
                await ws.send_text(msg)
            except Exception:
                dead.add(ws)  # dead connections removed lazily
        self.rooms[room_id] -= dead

manager = ConnectionManager()
redis_client: aioredis.Redis = None

@app.on_event("startup")
async def startup():
    global redis_client
    redis_client = aioredis.from_url("redis://localhost:6379")
    # Start Redis subscriber in background task
    asyncio.create_task(redis_subscriber())

async def redis_subscriber():
    """
    Listens to ALL Redis channels (ws:*) and fans out to local WebSocket connections.
    Each server instance runs this loop. When server A publishes to Redis,
    ALL servers receive it and forward to their local connections.
    """
    pubsub = redis_client.pubsub()
    await pubsub.psubscribe("ws:*")  # pattern subscribe: all ws: channels

    async for message in pubsub.listen():
        if message["type"] != "pmessage": continue
        channel = message["channel"].decode().removeprefix("ws:")
        data = message["data"].decode()
        await manager.broadcast_local(channel, data)

async def websocket_endpoint(ws: WebSocket, room_id: str):

    try:
        while True:
            data = await ws.receive_text()
            payload = json.loads(data)

            # Publish to Redis — ALL server instances will fanout to their clients.
            # This is the key: Redis decouples servers from knowing about each other.
            await redis_client.publish(f"ws:{room_id}", json.dumps({
                "room": room_id,
                "payload": payload,
            }))

    except WebSocketDisconnect:
        manager.leave(room_id, ws)

Deep-Dive Questions
You have 1 million concurrent WebSocket connections. Walk me through the architecture.
Each server can hold ~50-100k connections (memory-bound at ~4-8KB/connection for kernel buffers, event loop overhead). So you need 10-20 servers. Key design decisions: (1) Stateless message routing via Redis Pub/Sub — no sticky sessions needed, any server can accept reconnections. How do you handle WebSocket reconnections without losing messages?
The client must track the last message sequence number it received. On reconnect, it sends that sequence number. The server buffers the last N seconds of messages in Redis (LPUSH with LTRIM) or a message queue (Kafka with consumer offsets). The server replays any messages the client missed. This is the pattern Discord uses with their "resume" opcode. The window for buffering is a product decision: larger buffer = more storage cost, smaller buffer = higher risk of permanent loss on long disconnections. Kafka is ideal here because consumer offset tracking is built-in.

SSE — Server-Sent Events

Professor's Lecture

SSE is the simplest server-push mechanism: it's just an HTTP response that never ends. The Content-Type: text/event-stream tells the browser to treat the response as a stream of events. The wire format is dead simple — newline-delimited text fields. Browsers have a built-in EventSource API that automatically reconnects with the Last-Event-ID header, enabling at-least-once delivery semantics without any additional machinery.

SSE is vastly underused in industry. If your use case is server-to-client only (notifications, live dashboards, LLM streaming, stock tickers, order status), SSE is simpler, more scalable, and more proxy-friendly than WebSocket. It goes over plain HTTP so CDNs, proxies, and load balancers handle it without special configuration. HTTP/2 SSE is even better — you can have multiple event streams multiplexed on a single connection.

The limitation is directionality: only server can send. If the client needs to send data, it makes a separate HTTP request. This is actually an advantage for auditing, rate limiting, and middleware — the client's requests go through normal HTTP infrastructure while the server's events stream over SSE.

SSE Wire Format (text/event-stream):

  HTTP/1.1 200 OK
  Content-Type: text/event-stream
  Cache-Control: no-cache
  Connection: keep-alive
  X-Accel-Buffering: no          ← disable Nginx buffering!

  id: 1                          ← Last-Event-ID for reconnect resume
  event: price-update            ← custom event type (optional)
  data: {"symbol":"AAPL","price":189.42}
  retry: 3000                    ← tell client to wait 3s before reconnect
                                 ← blank line = event boundary
  id: 2
  data: {"symbol":"GOOG","price":141.87}

  :heartbeat                     ← comment line (: prefix) = keepalive, not an event

Client auto-reconnect:
  On disconnect → browser waits `retry` ms → sends:
  GET /events HTTP/1.1
  Last-Event-ID: 2               ← server can replay missed events

from fastapi import FastAPI, Request
from fastapi.responses import StreamingResponse
import asyncio
import json
import time
from collections import deque

app = FastAPI()

# ─── Event Buffer for replay-on-reconnect ─────────────────────────────────────
class EventBuffer:
    """Ring buffer of recent events. On reconnect, replay events after last_event_id."""
    def __init__(self, maxlen=1000):
        self.buffer: deque = deque(maxlen=maxlen)
        self.counter = 0

    def add(self, event_type: str, data: dict) -> dict:
        self.counter += 1
        event = {"id": self.counter, "event": event_type, "data": data}
        self.buffer.append(event)
        return event

    def since(self, last_id: int) -> list:
        """Return all events after last_id. Empty list if fully caught up."""
        return [e for e in self.buffer if e["id"] > last_id]

event_buffer = EventBuffer()
subscribers: set[asyncio.Queue] = set()

def format_sse(event: dict) -> str:
    """Format event as SSE wire format."""
    lines = [
        f"id: {event['id']}",
        f"event: {event['event']}",
        f"data: {json.dumps(event['data'])}",
        "",  # blank line = event boundary
    ]
    return "\n".join(lines)

@app.get("/events")
async def event_stream(request: Request):
    # Read Last-Event-ID for reconnect resume
    last_event_id = int(request.headers.get("last-event-id", 0))
    queue: asyncio.Queue = asyncio.Queue(maxsize=100)  # bounded = backpressure
    subscribers.add(queue)

    async def generate():
        try:
            # 1. Replay missed events on reconnect
            for missed in event_buffer.since(last_event_id):
                yield format_sse(missed)

            # 2. Send heartbeat every 15s to prevent proxy timeouts
            #    Many proxies (nginx, AWS ALB) will close idle connections.
            #    A comment line keeps the connection alive without creating an event.
            heartbeat_task = asyncio.create_task(_heartbeat(queue))

            while True:
                # Check if client disconnected
                if await request.is_disconnected():
                    heartbeat_task.cancel()
                    break

                try:
                    event = await asyncio.wait_for(queue.get(), timeout=20.0)
                    if event is None:  # heartbeat sentinel
                        yield ":heartbeat\n\n"
                    else:
                        yield format_sse(event)
                except asyncio.TimeoutError:
                    yield ":keepalive\n\n"

        finally:
            subscribers.discard(queue)

    async def _heartbeat(q: asyncio.Queue):
        while True:
            await asyncio.sleep(15)
            try:
                q.put_nowait(None)  # None = heartbeat signal
            except asyncio.QueueFull:
                pass  # slow consumer — drop heartbeat, not event

    return StreamingResponse(
        generate(),
        media_type="text/event-stream",
        headers={
            "Cache-Control": "no-cache",
            "X-Accel-Buffering": "no",     # CRITICAL: disable Nginx buffering
            "Access-Control-Allow-Origin": "*",
        }
    )

async def publish_event(event_type: str, data: dict):
    """Publish an event to all connected SSE clients."""
    event = event_buffer.add(event_type, data)
    dead = set()
    for q in subscribers:
        try:
            q.put_nowait(event)  # non-blocking: drop slow consumers
        except asyncio.QueueFull:
            dead.add(q)  # slow consumer — could also send error event
    subscribers -= dead
Deep-Dive Questions
When would you choose SSE over WebSocket?
SSE when: (1) you only need server-to-client push — news feeds, notifications, LLM token streaming, order status, live dashboards; (2) infrastructure simplicity matters — SSE is plain HTTP, every proxy/CDN/load balancer handles it without configuration; (3) automatic reconnection is important — EventSource handles this natively including Last-Event-ID replay; (4) you want per-connection auth via HTTP headers (WebSocket upgrade request supports headers, but the persistent frame channel has no native HTTP auth). WebSocket when: bidirectional low-latency is required — chat, gaming, collaborative editing, trading terminals where the client sends high-frequency data.
How would you handle 100,000 SSE connections on a single FastAPI server?
The async model helps enormously — each SSE connection is a coroutine, not a thread. 100k coroutines sharing one event loop is feasible. Key concerns: (1) Memory — ~2-4KB per connection for queue + coroutine stack. 100k × 4KB = ~400MB, acceptable. (2) Event fan-out — iterating 100k queues on every publish is O(N). Solution: use asyncio.gather() for parallel puts, or a pub/sub tree. (3) Queue backpressure — bound queue size and drop slow consumers rather than letting them drag down the whole server. (4) Nginx buffering — must set X-Accel-Buffering: no or Nginx buffers the entire response before forwarding. (5) At truly large scale, separate the SSE gateway from the application logic — Nginx can proxy SSE streams, keeping app servers stateless.

gRPC — Contract-First RPC over HTTP/2

Professor's Lecture

gRPC is not just a protocol — it's an RPC framework with strong opinions. You define your service in Protocol Buffers (.proto files), generate client and server stubs in any supported language, and the framework handles serialization, multiplexing, and load balancing. The contract is explicit and typed, which makes gRPC excellent for internal service-to-service communication where you control both sides.

The performance advantage over REST+JSON is real but often overstated. Protobuf encoding is 3-10x smaller than JSON and 5-7x faster to parse. But more importantly, gRPC gives you bidirectional streaming: a client can open a stream and send requests, the server sends responses — all on a single HTTP/2 connection, multiplexed with other calls. This enables patterns impossible with REST: server streaming (long-running queries), client streaming (chunked upload), and bidirectional streaming (interactive sessions).

The critical operational downside: gRPC is not natively browser-compatible. HTTP/2 trailers (used for gRPC status codes) are not accessible from browser JavaScript. gRPC-Web is the workaround, but it requires an Envoy proxy to translate. For browser clients, REST or SSE/WebSocket remain the right choice.

Deep-Dive Questions
When would you choose gRPC over REST for an internal service, and when would you not?
Choose gRPC when: (1) multiple language services need to interoperate with strong typing guarantees; (2) you need bidirectional streaming (e.g., real-time data pipelines, event sourcing, collaborative apps); (3) performance at the margin matters — Protobuf's binary encoding and HTTP/2 multiplexing measurably help under high RPS; (4) you want client SDK generation to be automatic and versioned. Avoid gRPC when: (1) browser clients are consumers — use REST or GraphQL; (2) the team is small and the overhead of .proto files + codegen adds more friction than value; (3) you need human-readable payloads for debugging/auditing; (4) your infrastructure doesn't support HTTP/2 end-to-end — many AWS services and API gateways don't support gRPC natively.
How does gRPC handle load balancing differently from REST?
REST over HTTP/1.1 is request-level — each request can go to a different backend, and L7 load balancers handle this trivially. gRPC over HTTP/2 maintains long-lived connections with many streams. If you put a TCP load balancer in front (L4), all streams on a connection go to the same backend — you lose load balancing. You need L7 load balancing that understands HTTP/2 frames (Envoy, nginx, Linkerd). Alternatively, client-side load balancing (etcd/Consul service discovery, the client picks a backend per RPC) eliminates the load balancer entirely, which is what gRPC's built-in client-side LB supports. grpc.max_connection_age_ms forces connection recycling, spreading load when new connections form.

  └── SSE: breaks when Nginx buffers responses (fix: X-Accel-Buffering: no)
  └── gRPC: breaks with L4 load balancers (fix: L7 LB or client-side LB)

Reliability Patterns

Professor's Lecture

Reliability at the protocol level means understanding exactly what guarantees each layer provides, and what gaps your application must fill. TCP gives you ordered, reliable byte delivery — but not message boundaries, not application-level acknowledgment, and not protection against process crashes. If your process dies after receiving bytes but before processing them, TCP's guarantee is worthless. You need application-level acknowledgments and idempotency.

The three axes of reliability: durability (data survives failures), availability (system responds despite partial failures), and consistency (responses reflect the true state). These trade off — pick which ones matter for each operation, and be explicit about the others.

# Problem: client retries a POST (charge $100), server receives it twice.
# Solution: client generates a UUID for each logical operation.
#   Server stores result keyed by idempotency key.
#   On retry, server returns the STORED result — never executes twice.
#
# This is exactly how Stripe's API works.

@app.post("/payments")
async def create_payment(
    body: dict,
    idempotency_key: Optional[str] = Header(None)
):
    if not idempotency_key:
        idempotency_key = str(uuid.uuid4())

    cache_key = f"idem:{idempotency_key}"

    # Check if we've already processed this key
    cached = r.get(cache_key)
    if cached:
        return json.loads(cached)  # return exact same response as first call

    # Process (this may take time and resources)
    result = process_payment(body)

    # Store result with TTL (24h is common — covers retry windows)
    r.setex(cache_key, 86400, json.dumps(result))
    return result

The Reliability Checklist For any network operation: (1) Is it idempotent? If not, add idempotency keys. (2) What happens on timeout — did the server receive the request or not? Add correlation IDs and status-check endpoints. (3) What's the retry strategy — exponential backoff + jitter, bounded retry count, circuit breaker. (4) Where is state stored — can you reconstruct it after a crash? (5) Is there a DLQ for unprocessable messages? (6) What's the monitoring story — how do you know it's failing?
Deep-Dive Questions
Explain exactly-once delivery semantics. Is it achievable?
True exactly-once across independent systems is impossible (see the Two Generals Problem). What we can achieve is exactly-once processing with idempotent consumers: deliver at-least-once, deduplicate at the consumer. Kafka's "exactly-once semantics" (since 0.11) achieves this within the Kafka cluster using producer idempotence and transactional APIs — the producer is assigned a PID, sequence numbers detect duplicates, transactions span multiple partitions atomically. But across system boundaries (Kafka → your DB → external API), you still need idempotency at each hop. The honest answer: "exactly-once" in marketing usually means "at-least-once with idempotent consumers."
What's the difference between the Outbox pattern and dual-writes? Why does it matter?
Dual-write: write to DB, then write to message queue — two separate operations, no atomicity. If the process crashes between them, you have DB updated but no event (or vice versa). Outbox: write to DB and outbox table in a single transaction, guaranteed atomic. A separate poller reads the outbox and publishes. This provides at-least-once delivery with a single source of truth (the DB). The failure mode is well-defined: the poller might publish the same event twice (if it crashes after publishing but before marking as sent), so consumers must be idempotent. Change Data Capture (Debezium) is an alternative that reads the DB write-ahead log instead of an outbox table.

Scalability Patterns

Professor's Lecture

Scalability is the ability to handle more load by adding resources, and the enemy of scalability is state. Stateless services scale trivially — add a server, route traffic to it, done. Stateful services (WebSocket connections, sessions, caches) require coordination: which server holds which state, how do you redistribute it when a server leaves, how do you avoid a single point of contention?

The key insight for protocol scalability: vertical protocols (one long-lived connection handles many operations) require careful state externalization. Horizontal protocols (one connection, one operation, then close) are stateless by nature. HTTP/1.1 REST leans horizontal. WebSocket leans vertical. gRPC is in between — long-lived connection, but state is minimal if you use the right patterns.

import asyncio
import aiohttp
import time
from collections import deque

# ─── Token Bucket Rate Limiter ────────────────────────────────────────────────
# Controls the rate of outbound requests or inbound processing.
# Token bucket allows bursts up to `capacity`, sustained rate of `rate/s`.

class TokenBucket:
    def __init__(self, capacity: int, rate: float):
        self.capacity = capacity    # max burst
        self.rate = rate            # tokens per second
        self.tokens = capacity
        self.last = time.monotonic()
        self._lock = asyncio.Lock()

    async def acquire(self, tokens=1) -> float:
        """Returns wait time in seconds. 0 means immediately available."""
        async with self._lock:
            now = time.monotonic()
            elapsed = now - self.last
            self.tokens = min(self.capacity, self.tokens + elapsed * self.rate)
            self.last = now

            if self.tokens >= tokens:
                self.tokens -= tokens
                return 0.0
            else:
                wait = (tokens - self.tokens) / self.rate
                return wait

# ─── Adaptive Concurrency Limiter ────────────────────────────────────────────
# Inspired by Netflix's adaptive concurrency limiting (TCP Vegas algorithm).
# Adjusts inflight request limit based on observed latency.
# Prevents overloading downstream services that can't reject fast enough.

class AdaptiveConcurrencyLimiter:
    def __init__(self, initial_limit=10, min_limit=1, max_limit=200):
        self.limit = initial_limit
        self.min_limit = min_limit
        self.max_limit = max_limit
        self.inflight = 0
        self.min_latency = float('inf')  # RTT with no queuing
        self._lock = asyncio.Lock()

    async def __aenter__(self):
        async with self._lock:
            while self.inflight >= self.limit:
                await asyncio.sleep(0.01)
            self.inflight += 1
        self._start = time.monotonic()
        return self

    async def __aexit__(self, exc_type, *_):
        latency = time.monotonic() - self._start
        self.min_latency = min(self.min_latency, latency)
        gradient = 1 - (self.min_latency / latency) if latency > 0 else 0

        async with self._lock:
            self.inflight -= 1
            if not exc_type and gradient > 0.25:  # latency growing: reduce limit
                self.limit = max(self.min_limit, int(self.limit * 0.9))
            elif not exc_type:                        # healthy: increase limit
                self.limit = min(self.max_limit, self.limit + 1)

# ─── Consistent Hashing for stateful connection routing ──────────────────────
# Used by WebSocket gateways, cache servers (memcached, Redis Cluster).
# Ensures the same key routes to the same node with minimal remapping on resize.

import bisect

class ConsistentHashRing:
    def __init__(self, nodes: list[str], replicas=150):
        """
        replicas: virtual nodes per physical node.
        More replicas = better distribution, more memory.
        150 is a common production value.
        """
        self._ring: dict[int, str] = {}
        self._sorted_keys: list[int] = []
        for node in nodes:
            self.add_node(node, replicas)

    def _hash(self, key: str) -> int:
        return int(hashlib.md5(key.encode()).hexdigest(), 16)

    def add_node(self, node: str, replicas=150):
        for i in range(replicas):
            h = self._hash(f"{node}:{i}")
            self._ring[h] = node
            bisect.insort(self._sorted_keys, h)

    def remove_node(self, node: str, replicas=150):
        for i in range(replicas):
            h = self._hash(f"{node}:{i}")
            del self._ring[h]
            self._sorted_keys.remove(h)

    def get_node(self, key: str) -> str:
        """Route key to node. Same key always goes to same node."""
        if not self._ring: raise ValueError("Ring is empty")
        h = self._hash(key)
        idx = bisect.bisect(self._sorted_keys, h) % len(self._sorted_keys)
        return self._ring[self._sorted_keys[idx]]

Deep-Dive Questions
Design a system to deliver real-time notifications to 50 million users. Walk me through your protocol choices and scaling strategy.
Tier 1 — Connection Layer: SSE over HTTP/2 (not WebSocket — notifications are unidirectional, SSE is simpler and proxies handle it natively). Each server holds ~50k connections; need ~1000 servers. Consistent hashing routes each user_id to a deterministic server. Tier 2 — Fanout Layer: When an event is generated, look up which servers hold connections for the target users. At 50M users this lookup must be fast — Redis hash of user_id → server_id. Publish to the specific server's Redis channel. For broadcast events (everyone gets it), use a Kafka topic partitioned by server_id — each server consumes its partition. Tier 3 — Backpressure: Slow consumers get their SSE queue dropped when full. The client's EventSource reconnects, catches up via Last-Event-ID, and misses any events outside the buffer window. For guaranteed delivery, persist events in Cassandra (time-series, user-partitioned) and replay on reconnect. The key architecture decision: separate the connection-holding tier from the business logic tier. The connection servers are dumb pipes; app servers are stateless.
What is the thundering herd problem in the context of connection scaling, and how do you mitigate it?
Thundering herd: a large number of clients simultaneously try to reconnect after a server restart or network partition. Example: 100k WebSocket connections drop when a server restarts. All 100k try to reconnect simultaneously, overwhelming the server (or all its peers). Mitigations: (1) Randomized reconnect delay — clients wait random(0, base_delay * 2^attempt) before reconnecting, spreading the load across seconds. (2) Rolling restarts — restart one server at a time with health check gates, limiting the number of simultaneous disconnections. (3) Connection admission control — rate limit new connections at the load balancer (e.g., Nginx's limit_conn directive). (4) Sticky session groups — when one server is down, its clients are gradually redistributed rather than all hitting one new server. This is why microservice resilience patterns like jitter exist: they're all solving variations of the same thundering herd problem.
How does backpressure propagate through an async system, and what happens when it doesn't?
Backpressure is a signal from consumer to producer saying "slow down." In async systems it propagates through: (1) bounded queues — when a queue is full, the producer blocks or drops; (2) HTTP/2 flow control — the receiver sets a window; the sender blocks when it's exhausted; (3) TCP's receive window — the kernel blocks send() when the receiver's buffer is full. When backpressure breaks down: producers write faster than consumers can read. Unbounded queues grow until OOM. Memory exhaustion causes the entire process to crash. Classic pattern in stream processing — the ingestion pipeline absorbs a spike while the processing stage is at capacity, the queue grows, OOM kills the process, all buffered data is lost. Solutions: bounded queues with explicit drop policies (drop-head, drop-tail, random), reactive streams protocols (Project Reactor, RxJava have backpressure operators built in), or external queues (Kafka) where the "queue" is disk, not RAM.
